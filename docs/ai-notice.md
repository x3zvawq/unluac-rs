# AI Notice

这份文档面向会使用 AI Agent 维护这个仓库的开发者。

当前仓库的核心文档入口如下：

- 设计文档见 [design.md](./design.md)，当你刚开始接手这个项目的时候务必阅读。
- 调试文档见 [debug.md](./debug.md)，排查错误时务必阅读。

## 工作时一定、必须遵循的规则

- 务必使用以下skills: rust-best-practices
- 当遇到需要在几个选项中做选择的时候，直接列出当前的几个选项，并且讲清楚每一个选项的优劣之后，让用户决定选用什么方式解决。
- 请注意，当你在某个case下发现问题的时候，应该尽可能追踪是不是更早的某个层次漏掉了什么东西，就算找到了觉得是问题的根源的也不妨再多推几步确认推断。**不要** 在后面的层次为前面缺失或者错误的东西兜底。
- **永远不要** 想着用堆特判的方式去让某个问题不再出现，而是应该在遇到问题的时候定位问题的根源尝试从结构上去修复问题。判定规则是永远都写不完的，再多的判定规则也总是会被特定的case hack。
- 每一个文件头都需要通过文档注释的方式解释当前文件的的设计理念，也就是说为啥要要有这个文件？这个文件提供什么方法或者struct？对了，使用中文注释。
- 在代码可能产生误解或者代码本身不足以解释“为什么”的时候也需要补充中文注释，告知维护者某一段代码为什么存在，拒绝翻译代码的注释，也就是说你需要做的是讲清楚“为什么”而非“做什么”。
- 当某个文件超过1000行的时候，需要特别注意是否可以根据职责 或者 根据关注点进行拆分，当然如果确实职责一致且关注点一致的话可以保留长度过长的文件。
- 该库需要考虑到编译成wasm模块的方式分发，因此需要避免使用某些导致库无法编译为wasm模块的操作或者依赖。
- 每一轮工作完成如果有发生代码库的变化的话，需要额外输出一条在代码块内的git提交记录，以说明当前轮次做了什么内容，并且应该按照`<type>(<scope>): <subject>`的规则，比如: `fix(ast): 修复短路表达式的性能黑洞`
