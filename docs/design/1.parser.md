# Parser

## 目标

Parser 负责把某个 dialect/version 的 Lua 字节流解析成保真的原始程序表示。

解析流程：

1. `Header Parse`
2. `Proto Parse`
3. `Instruction Decode`

## 入口

- 每个 dialect 单独实现一个 parser struct
- 总入口通过 `Adapter/Resolver` 选择具体 parser
- 自动探测失败直接报错，由用户显式指定 dialect/version

## Parse Options

```text
ParseOptions
  mode: strict | permissive
  string_encoding
  string_decode_mode: strict | lossy
```

规则：

- `strict` 下 header、magic number、布局、索引、opcode 异常直接报错
- `permissive` 下只要还能继续解析就继续，只有解析无法继续时才在 parser 层报错
- `string_encoding` 只用于生成字符串的可选文本视图
- decode 失败不丢弃原始 bytes

## 关键决策

- `RawProto` 采用 `common + extra`
- `constants`、`upvalues`、`debug_info` 也采用 `common + extra`
- 所有 `extra` 都使用 enum，不使用 trait object
- `line_range` 使用统一命名 `defined_start / defined_end`
- `RawConstPoolCommon` 区分 `Integer` 和 `Number`
- dialect 特有常量类型放在 `DialectConstPoolExtra`
- child proto 不进入 `RawLiteralConst`，统一放在 `children`
- `RawInstr` 只存解码后的指令，不保留第二份 raw 指令数组
- 原始编码、偏移和 span 信息统一挂在 `origin`
- 字符串常量统一使用 `RawString`
- `RawString` 永远保留原始 `bytes`
- `RawString.text` 是按 parse options 生成的可选解码视图，不覆盖原始值
- `ProtoSignature` 必须保留后续结构恢复真正依赖的签名事实，而不只是一句“是否 vararg”
- parser 可以共享 reader 基础设施，但版本专属的 varint / string / extra-word 规则仍属于各自 dialect parser
- 像 `LOADKX / NEWTABLE / SETLIST + EXTRAARG` 这类扩展字编码需要在 parser 阶段折叠进前一条 raw 指令的 `extra`
- Lua 5.5 的字符串复用表属于 parser 契约，复用索引必须在 parser 阶段校验并还原成稳定 `RawString`

## 规则

- `Header Parse` 负责 dialect/version 探测
- `Proto Parse` 递归解析 proto 树
- `Instruction Decode` 在 parser 阶段立即执行
- 遇到未知 opcode 直接报错，不保留 unknown instruction 形态
- parser 保存 offset/span 与必要原始值
- 不同 dialect 不共享 raw instruction enum
- parser 可以共享读取基础设施，但不共享解析规则
- `Header Parse` 除了 magic/version 外，还负责各版本特有的 sentinel / layout / 数值编码校验
- parser 不允许把扩展字 `EXTRAARG` 作为游离 raw 指令泄漏给后层；如果某个 opcode 语义上依赖额外字，parser 必须在这一层把它和前一条指令重新绑定
- Lua 5.4 和 Lua 5.5 虽然都使用 varint 风格编码，但终止规则不同，不能共享同一套解码假设
- Lua 5.5 parser 需要处理字符串复用索引，把复用字符串恢复成正常 `RawString`，而不是把“reuse slot”泄漏给后层

## Consumes / Provides

- Consumes: 原始字节流、调用方给定的 parse options
- Provides: `RawChunk`, `ChunkHeader`, `RawProto`, `RawInstr`, `RawString`

## 后续依赖

Transformer 依赖 Parser 提供的这些事实：

- `ChunkHeader.dialect/version`
- `RawProto` 的 proto 树结构
- 常量池、upvalue、debug info
- 已解码的 `RawInstr`
- `Origin` 和字符串原始 bytes

这些信息后面不会重新回 parser 推断，因此 parser 阶段必须一次性保真保存。
