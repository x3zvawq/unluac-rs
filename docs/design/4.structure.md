# StructureFacts

## 目标

StructureFacts 只产出结构候选和结构约束，不直接产出最终源码结构结论。

这层的职责是把：

- low-IR 指令形状
- CFG 上的图事实
- dominator / loop / frontier 结果
- dataflow 的 def-use / phi / effect 信息

转成更贴近恢复工作的候选集合，交给 HIR 去做最终判定。

## 当前规则

- `StructureFacts` 默认基于 reachable 子图分析
- `StructureFacts` 消费 `LoweredProto.instrs`、`Cfg`、`GraphFacts` 和 `DataflowFacts`
- `StructureFacts` 负责识别可结构化候选与必须保留跳转的区域
- `StructureFacts` 产的是 hint、candidate、requirement，不是最终语法

## 边界

- `LoopCandidate.kind_hint` 只是提示，不是最终循环语法结论
- `WhileLike` 应优先保留给“header 自己就是纯条件入口”的 loop；如果 header 里已经混入主体语句，
  StructureFacts 不应为了追求 while 形状而提前吞掉这些前缀
- `RepeatLike` 允许 continue 条件后面再跟一层线性 backedge pad，只要这个 pad 只是把控制流送回
  header；这样 `repeat ... until` 不会因为 VM 里多了一块 jump/close 垫片就被误判成 unknown
- `ShortCircuitCandidate` 只是表达式形状控制流候选，不等于最终 `and/or`
- `RegionFact` 只描述区域事实，不直接决定 AST 结构
- `GotoRequirement` 明确标出无法被当前候选吸收的跳转
- `StructureFacts` 可以直接读取 low-IR 指令形状，但不负责重算 CFG / GraphFacts / Dataflow
- `BranchCandidate.merge = None` 表示这条分支没有在可结构化区域内重新汇合，常见于双端都直接终止的分支
- `ShortCircuitExit::ValueMerge` 和 `ShortCircuitExit::BranchExit` 分别表示“值合流型短路”和“条件出口型短路”
- `ShortCircuitNode.truthy/falsy` 已经按 Lua 求值语义归一化，不再暴露 raw `then/else`
- `ValueMerge` 候选允许内部出现共享 continuation，因此契约上使用受控 DAG，而不是线性链
- `BranchValueMergeCandidate` 只表达“这个结构化 branch 在 merge 点上合流了哪些 phi 值，以及每条 arm
  各自贡献了哪些 predecessor”，不直接决定最终要落成同一 lvalue、`Decision` 还是别的 HIR 形状
- `GotoReason::UnstructuredContinueLike` 表示“当前候选暂时还吞不掉一条 loop-local 的 continue 语义边”，
  不等价于最终源码必须保留 `goto`
- 只有“主动提前跳到 continue target”的控制边才应该落成 `UnstructuredContinueLike`；
  顺着线性 fallthrough 自然进入 continue block 的 body tail 仍然算 loop 的正常执行路径
- 这类 continue-like requirement 后续允许被 HIR 吸收成更高层的循环控制语义；是否最终打印成
  `continue`、还是回退成 `goto + label`，要留给 target-aware 的 AST 再拍板

## 各类候选的用途

### BranchCandidate

回答：

- 某个 branch 更像 `if-then`、`if-else` 还是 guard
- merge 点在哪里
- 为了得到更自然结构，条件是否值得反转

### LoopCandidate

回答：

- 某个自然循环候选更像前测循环、后测循环还是 `for` 形态
- continue/exit 候选在哪里
- 这是不是一个可规约循环
- 当前 `continue_target` 表示“本轮 loop body 结束后会回到哪一块继续判定下一轮”，
  它既可能就是 header，也可能是 header 之前的一块条件分支

### ShortCircuitCandidate

回答：

- 某片分支控制流是否更像“值产生型控制流”或“条件链控制流”
- 是否值得优先尝试恢复成短路表达式

如果是 `ValueMerge`：

- 说明短路链最终汇合到一个 merge block
- `result_reg` 可以作为值恢复的直接证据
- `nodes + truthy/falsy target` 用来表达共享 continuation，例如 `(a and b) or (c and d)`

如果是 `BranchExit`：

- 说明短路链直接把控制流导向 truthy/falsy 两个出口
- 这类候选主要服务 `if / while` 条件恢复，不要求存在 phi merge
- 当前实现对 `BranchExit` 仍然保持保守，优先覆盖线性条件链；值合流型 DAG 则允许更一般的共享形状

### GotoRequirement

回答：

- 哪些跳转无法被当前结构候选吸收
- 哪些 block 需要 label
- 哪些边必须保留成显式 goto

这块很关键，因为 HIR/AST 后续不应再重新猜“哪些地方必须保留跳转”。

但也要区分两类语义：

- `IrreducibleFlow / CrossStructureJump / MultiEntryRegion` 这类 requirement 更接近“必须保留显式跳转”
- `UnstructuredContinueLike` 更接近“当前还没被结构候选吸收的 loop-local 控制语义”

### RegionFact

回答：

- 某片 block 集合是否是单入口单出口区域
- 是否可规约
- 是否值得继续尝试结构化

### ScopeCandidate

回答：

- 哪些区域可能对应局部作用域边界
- 哪些 loop/branch/close 点会强制形成 scope 切分

### BranchValueMergeCandidate

回答：

- 某个 `if/else` merge 上的 phi，是否明确属于这条 branch 的两臂合流
- 每个 merge 值在 then / else 两臂分别来自哪些 predecessor

这条候选的作用不是把普通 branch 误判成短路，而是把“结构化 branch + 值合流”这层证据
前置表达出来，避免 HIR 再回头从 `BranchCandidate + PhiCandidate` 里反推一次。

## Consumes / Provides

- Consumes: `LoweredProto.instrs`, `Cfg`, `GraphFacts`, `DataflowFacts`
- Provides: `branch_candidates`, `branch_value_merge_candidates`, `loop_candidates`, `short_circuit_candidates`, `goto_requirements`, `region_facts`, `scope_candidates`

## 后续依赖

HIR 会直接依赖这些事实：

- 先尝试使用 `short_circuit_candidates` 恢复表达式形状控制流
- 再使用 `branch_candidates`、`branch_value_merge_candidates` 和 `loop_candidates` 做语句级结构恢复
- 使用 `goto_requirements` 决定哪些边不能再继续结构化
- 使用 `scope_candidates` 和 `Close` 相关点建立作用域边界

这样 HIR 的工作重点就变成：

- 在候选之间做最终决策
- 恢复表达式和值流
- 选择最自然的结构形式

而不是回过头再重算底层图事实。
