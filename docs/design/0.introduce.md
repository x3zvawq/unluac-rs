# 总览

## 文档用途

这份文档描述仓库的正式流水线、层间依赖、共享设施与维护总规则。

它回答的核心问题只有 4 个：

- 某个问题应该在哪一层解决
- 某一层可以依赖哪些前层事实
- 某一层应优先复用哪些共享设施
- 某一层不能越权做什么

## 正式流水线

1. Parser
2. Transformer
3. CFG
4. GraphFacts
5. Dataflow
6. StructureFacts
7. HIR
8. AST
9. Readability
10. Naming
11. Generate

`Debugging` 和 `Test` 与正式流水线并行存在，负责维护、排错和验证，不参与源码恢复本身。

## 每一层的职责摘要

- `Parser`：把字节流解析成保真的 raw 模型，并保存协议事实。
- `Transformer`：把 raw lowering 成统一 low-IR，并建立 low/raw 映射。
- `CFG`：把 low-IR 组织成 basic block 与控制流边。
- `GraphFacts`：从 CFG 产出支配、回边、自然循环等图事实。
- `Dataflow`：从 CFG 与图事实产出 def/use、phi、活跃性、副作用等数据流事实。
- `StructureFacts`：把图事实和数据流事实翻译成源码恢复候选。
- `HIR`：做恢复决策，输出语义上稳定、面向 AST 的中间表示。
- `AST`：把 HIR 落成合法 Lua 语法树，并保持目标方言约束。
- `Readability`：只做“更像源码”的整形，不补前层事实。
- `Naming`：给最终保留下来的绑定分配字符串名字。
- `Generate`：把 AST 与 NameMap 渲染成源码文本。

## 跨层维护规则

- 单向依赖：后层只能消费前层事实，不能反向侵入前层职责。
- 单一事实源：某个稳定事实一旦在前层显式保存，后层只应通过 query/accessor 消费它。
- 结构优先：不要用特判、静默 fallback 或“后层兜底”掩盖前层事实缺失。
- 共享实现优先：有共享 helper、macro、walker、visitor 时，应先接入共享层。
- 查询优先：后层需要稳定图查询或数据流查询时，优先把 query 补到事实层，而不是各自扫描容器。
- 输出层纯粹：`Readability`、`Naming`、`Generate` 不承担前层恢复失败的补救职责。

## 层间依赖图

- `Parser` 提供 `RawChunk`、`RawProto`、`RawInstr`、各 dialect raw extra。
- `Transformer` 消费 parser raw，并提供 `LoweredChunk`、`LoweringMap`、`LowInstr`。
- `CFG` 消费 `LoweredChunk`，提供 `Cfg` 与 block/edge 查询。
- `GraphFacts` 消费 `Cfg`，提供 dominator/post-dominator/backedge/natural-loop 等图事实。
- `Dataflow` 消费 `LoweredChunk + Cfg + GraphFacts`，提供 def/use/phi/effect/liveness/open-pack 等数据流事实。
- `StructureFacts` 消费 `LoweredChunk + Cfg + GraphFacts + Dataflow`，提供 branch/loop/short-circuit/goto/scope 等结构候选。
- `HIR` 消费 `LoweredChunk + Cfg + GraphFacts + Dataflow + StructureFacts`，提供结构化语义树。
- `AST` 消费 `HIR + AstTargetDialect`，提供合法 Lua 语法树。
- `Readability` 消费 `AST + ReadabilityOptions`，提供稳定、可读的 AST。
- `Naming` 消费 `AST + NamingEvidence`，提供 `NameMap`。
- `Generate` 消费 `AST + NameMap + GenerateOptions`，提供源码文本。

## 性能与可维护性约束

- 主路径优先使用稳定索引与 `Vec`/稠密结构。
- 主路径避免重复 BFS、重复 `BTreeMap/BTreeSet` 扫描和 clone 扩散。
- 图查询、数据流查询、binding 查询尽量集中到共享层。
- 新增共享设施时，优先放在真正拥有该事实的层，而不是放在消费者层。
- 每一正式层都应能接入统一 dump 与 timing。

## 处理问题的推荐顺序

当某个 case 暴露异常时，应按下面顺序排查：

1. 该事实是否已经在更前层存在，但后层没有复用。
2. 该事实是否应该在更前层显式保存，而不是后层回推。
3. 该问题是否只是共享 helper 未被复用。
4. 只有在前层事实已经充分、共享设施也正确复用后，才在所在层调整业务规则。
