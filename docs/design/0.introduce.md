# 总览

> 本章回答 4 个问题：某个问题该在哪一层解决、可以依赖哪些前层事实、该复用哪些共享设施、不能越权做什么。

## 正式流水线

```text
Parse → Transform → CFG → GraphFacts → Dataflow → StructureFacts → HIR → AST → Readability → Naming → Generate
```

`Debugging` 和 `Test` 与正式流水线并行存在，负责维护、排错和验证，不参与源码恢复。

### Pipeline 调度入口

| 文件 | 职责 |
|---|---|
| `src/decompile/pipeline.rs` | 主入口 `decompile(bytes, options)`，线性推进各阶段，收集 debug output 与 timing |
| `src/decompile/state.rs` | `DecompileStage` 枚举（Parse~Generate）+ `DecompileState` 状态容器（每层产物一个 `Option<T>` 槽位） |
| `src/decompile/contracts.rs` | 层间类型别名：`LoweredChunk`、`CfgGraph`、`GraphFacts`、`DataflowFacts`、`StructureFacts`、`HirChunk`、`AstChunk`、`ReadabilityResult`、`NamingResult`、`GeneratedChunk` |
| `src/decompile/options.rs` | `DecompileOptions`：dialect、parse、debug、readability、naming、generate 选项 |
| `src/decompile/debug.rs` | `DebugOptions` + `StageDebugOutput` + `define_stage_dump!` 宏 + `collect_stage_dump` |

## 每一层的职责摘要

| 层 | 职责 | 主入口 | 产物 |
|---|---|---|---|
| Parser | 字节流 → 保真 raw 模型 | `parse_chunk` | `RawChunk` |
| Transformer | raw → 统一 low-IR + `LoweringMap` | `lower_chunk` | `LoweredChunk` |
| CFG | low-IR → basic block + 控制流边 | `build_cfg_proto` | `CfgGraph` |
| GraphFacts | CFG → 支配 / 回边 / 自然循环 | `analyze_graph_facts` | `GraphFacts` |
| Dataflow | CFG + GraphFacts → def/use / phi / liveness / side-effect | `analyze_dataflow` | `DataflowFacts` |
| StructureFacts | 图 + 数据流 → 源码恢复候选 | `analyze_structure` | `StructureFacts` |
| HIR | 做恢复决策 → 语义树 | `analyze_hir` | `HirModule` |
| AST | HIR → 合法 Lua 语法树 | `lower_ast` | `AstModule` |
| Readability | AST 整形 → 更像源码的稳定 AST | `make_readable` | `AstModule` |
| Naming | 绑定身份 → 字符串名字 | `assign_names_with_evidence` | `NameMap` |
| Generate | AST + NameMap → 源码文本 | `generate_chunk` | `GeneratedChunk` |

## 层间依赖图

```text
Parser ─→ RawChunk
Transformer ─→ LoweredChunk, LoweringMap       (消费 RawChunk)
CFG ─→ Cfg, CfgGraph                           (消费 LoweredChunk)
GraphFacts ─→ DominatorTree, NaturalLoop...     (消费 Cfg)
Dataflow ─→ DataflowFacts                       (消费 LoweredChunk + Cfg + GraphFacts)
StructureFacts ─→ BranchCandidate, LoopCandidate...  (消费 LoweredChunk + Cfg + GraphFacts + Dataflow)
HIR ─→ HirModule                                (消费 LoweredChunk + CfgGraph + GraphFacts + Dataflow + StructureFacts)
AST ─→ AstModule                                (消费 HirModule + AstTargetDialect)
Readability ─→ AstModule                        (消费 AstModule + ReadabilityOptions)
Naming ─→ NameMap                               (消费 AstModule + HirModule + NamingEvidence)
Generate ─→ GeneratedChunk                      (消费 AstModule + NameMap + GenerateOptions)
```

## 跨层维护规则

1. **单向依赖**：后层只消费前层事实，不反向侵入前层职责。
2. **单一事实源**：某个稳定事实在前层显式保存后，后层只通过 query/accessor 消费。
3. **结构优先**：不要用特判、静默 fallback 或"后层兜底"掩盖前层事实缺失。
4. **共享实现优先**：有共享 helper / macro / walker / visitor 时先接入，再考虑新增。
5. **查询优先**：后层需要稳定图/数据流查询时，优先补到事实层，不各自扫描容器。
6. **输出层纯粹**：Readability / Naming / Generate 不承担前层恢复失败的补救。

## 处理问题的推荐顺序

当某个 case 暴露异常时：

1. 该事实是否已经在更前层存在，但后层没有复用。
2. 该事实是否应该在更前层显式保存，而不是后层回推。
3. 该问题是否只是共享 helper 未被复用。
4. 只有在前层事实已充分、共享设施也正确复用后，才在所在层调整业务规则。

## 性能约束

- 主路径优先使用稳定索引与 `Vec` / 稠密结构。
- 避免重复 BFS、重复 `BTreeMap/BTreeSet` 扫描和 clone 扩散。
- 图查询、数据流查询、binding 查询集中到共享层。
- 新增共享设施放在真正拥有该事实的层，不放在消费者层。
