# CFG, GraphFacts And Dataflow

这 3 层共享同一组底层模块，但职责严格分开：

- `CFG` 负责图结构
- `GraphFacts` 负责图分析事实
- `Dataflow` 负责 def/use、phi、活跃性与副作用

## CFG

### 职责

CFG 负责把线性 low-IR 组织成 basic block 与控制流边，并提供后层统一使用的 block/edge 查询。

### 入口与模块

- `src/cfg/build.rs`
- `src/cfg/common/cfg.rs`
- `src/cfg/common/storage.rs`
- `src/cfg/mod.rs`
  - `build_cfg_graph`

### 应消费的前层事实

- `LoweredProto.instrs`
- `LowInstr` 的控制流终结语义
- `LoweringMap` 的稳定索引关系

### 应优先复用的共享设施

`Cfg` 自己已经提供正式查询接口。后层如果需要 block/edge 信息，应优先补到 `Cfg`，而不是在 Structure 或 HIR 重写一份 helper。

典型查询包括：

- `branch_edges`
- `reachable_successors`
- `unique_reachable_successor`
- `reachable_predecessors`
- `can_reach`
- `can_reach_within`

### 维护规范

- 切块规则只放在 CFG。
- edge kind 语义只放在 CFG。
- 如果后层需要稳定 block/edge 查询，优先回到 CFG 补 query。

### 向后提供的事实

- `Cfg`
- `CfgGraph`
- `BasicBlock`
- `BlockRef`
- `CfgEdge`
- `EdgeKind`

## GraphFacts

### 职责

GraphFacts 负责从 CFG 计算稳定图事实。

### 入口与模块

- `src/cfg/graph.rs`
- `src/cfg/common/graph.rs`
- `src/cfg/mod.rs`
  - `analyze_graph_facts`

### 应消费的前层事实

- `Cfg`

### 应优先复用的共享设施

GraphFacts 的公开契约应优先表现为 query，而不是要求后层直接依赖具体容器形状。

已经存在的共享点包括：

- dominator / post-dominator tree
- backedges
- natural loops
- dominance frontier
- 相关 ancestor / dominates 查询

### 维护规范

- 同一类图分析只算一次，再派生到多个事实。
- 后层频繁需要的稳定图结论，应补成 GraphFacts query 或共享事实。
- 不要让 Structure/HIR 自己再做一份 dominator、frontier、region 公共分析。

### 向后提供的事实

- `GraphFacts`
- `DominatorTree`
- `PostDominatorTree`
- `NaturalLoop`

## Dataflow

### 职责

Dataflow 负责把 CFG 与 GraphFacts 上的控制流关系翻译成可恢复源码所需的数据流事实。

### 入口与模块

- `src/cfg/dataflow.rs`
- `src/cfg/dataflow/effects.rs`
- `src/cfg/dataflow/fixed.rs`
- `src/cfg/dataflow/liveness.rs`
- `src/cfg/dataflow/phi.rs`
- `src/cfg/dataflow/values.rs`
- `src/cfg/common/dataflow.rs`
- `src/cfg/mod.rs`
  - `analyze_dataflow`

### 应消费的前层事实

- `LoweredProto`
- `Cfg`
- `GraphFacts`

### 应优先复用的共享设施

DataflowFacts 自己已经承接了一批正式查询。后层如果开始直接扫描底层容器，优先把查询补回这里。

典型查询包括：

- `def_reg`
- `def_block`
- `def_instr`
- `instr_def_for_reg`
- `latest_local_def_in_block`
- `phi_use_count`
- `phi_used_only_in_block`

### 维护规范

- `DefId`、`PhiId`、open-pack 相关事实应集中在 DataflowFacts。
- HIR/Structure 不应直接把 `dataflow.defs`、`use_values`、`phi_candidates` 当成裸存储结构来扫。
- 如果后层频繁需要“某类 phi/def/use 的语义化查询”，应优先补到 DataflowFacts。
- 热路径优先使用稠密结构、轻量集合和共享索引，不在后层重复 clone。

### 向后提供的事实

- `DataflowFacts`
- `DefId`
- `PhiId`
- `InstrEffect`
- `SideEffectSummary`
- `InstrReachingDefs`
- `InstrReachingValues`
- `InstrUseDefs`
- `InstrUseValues`
- `PhiCandidate`

## 这三层的共同维护规则

- 结构层与 HIR 层只能通过公开 query/事实消费这三层结果。
- 若某个查询已经被多个消费者重复实现，应优先回收到 `src/cfg/common/`。
- `common/storage.rs` 负责紧凑容器；如果某个结构只是主路径性能设施，应优先放这里，而不是散落到业务层。
