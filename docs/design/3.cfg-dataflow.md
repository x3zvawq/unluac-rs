# CFG, GraphFacts And Dataflow

## CFG

### 当前规则

- CFG 基于经典 basic block 构建
- leader 只按控制流切分，不按语义切分
- 每个 block 最多一个 terminator，且 terminator 只能出现在 block 尾
- CFG 边按产生方式区分 kind
- CFG 将普通 terminator 与结构化 loop terminator 一视同仁地视为 block 结束点
- CFG 层把 `InstrRef` 跳转目标提升为 `BlockRef`
- CFG 不引入 synthetic entry
- CFG 引入一个 synthetic exit，用于统一 `Return` 和 `TailCall` 出口
- CFG 输出附带基础索引，供后续分析复用

### 切块规则

leader 包括：

- 第一条指令
- 所有跳转目标
- 所有 terminator 的下一条指令

### 边类型

```text
EdgeKind
  Fallthrough
  Jump
  BranchTrue
  BranchFalse
  LoopBody
  LoopExit
  Return
  TailCall
```

规则：

- `backedge`、`loop edge`、`irreducible edge` 不在 CFG 原生边类型中表示
- 这些信息由后续分析计算
- `LoopBody / LoopExit` 仅表示 low-IR loop terminator 直接产生的边

### 数据结构

```text
Cfg
  blocks: Vec<BasicBlock>
  edges: Vec<CfgEdge>
  entry_block: BlockRef
  exit_block: BlockRef
  block_order
  instr_to_block
  preds
  succs
  reachable_blocks

EdgeRef
  index

BlockRef
  index

BasicBlock
  kind: BlockKind
  instrs: InstrRange

BlockKind
  Normal
  SyntheticExit

InstrRange
  start: InstrRef
  len

CfgEdge
  from: BlockRef
  to: BlockRef
  kind: EdgeKind
```

规则：

- `entry_block` 是第一条指令所在 block 的真实入口
- `exit_block` 是唯一 synthetic exit block
- synthetic exit block 不承载真实指令
- `preds` 和 `succs` 保存 `EdgeRef`
- `reachable_blocks` 记录从 `entry_block` 可达的 block 集合
- `block_order` 只记录真实 block 的线性顺序，不包含 synthetic exit
- block 不缓存 terminator，按 `InstrRange` 尾指令即时判断

### 出口规则

- 每个 `Return` block 连接一条 `Return` 边到 `exit_block`
- 每个 `TailCall` block 连接一条 `TailCall` 边到 `exit_block`
- 存在多个 `Return/TailCall` 时，统一汇聚到同一个 `exit_block`
- 后续 post-dominator 和 exit-sensitive 分析都以 `exit_block` 为统一出口
- unreachable 的 `Return/TailCall` block 仍然保留这些出口边

### 可达性规则

- CFG 保留完整 block/edge 图，不在构图阶段删除 unreachable block
- `reachable_blocks` 是主分析默认使用的子图范围
- 诊断、调试和异常输入分析可以访问完整 CFG

### Consumes / Provides

- Consumes: `LoweredProto.instrs`
- Provides: `Cfg`, `entry_block`, `exit_block`, `reachable_blocks`, `instr_to_block`, `preds`, `succs`

## GraphFacts

### 当前规则

- `GraphFacts` 是基于 `Cfg` 的第一层派生图分析
- `GraphFacts` 只产出图事实，不产出源码级结构结论
- 所有图分析默认基于 `reachable_blocks` 子图计算
- `post_dominator` 统一以 CFG 的 `exit_block` 为出口

### 数据结构

```text
GraphFacts
  rpo
  dominator_tree: DominatorTree
  post_dominator_tree: PostDominatorTree
  dominance_frontier
  backedges
  loop_headers
  natural_loops

DominatorTree
  parent
  children
  order

PostDominatorTree
  parent
  children
  order

NaturalLoop
  header: BlockRef
  backedge: EdgeRef
  blocks
```

规则：

- `rpo` 是 reachable 子图上的 reverse post order
- `parent + children + order` 同时保留，避免后续分析重复构建
- `dominance_frontier` 作为图事实直接保留，供 SSA-like 和 phi candidate 分析使用
- `backedges`、`loop_headers` 和 `natural_loops` 都是图事实，不代表最终可结构化循环

### Consumes / Provides

- Consumes: `Cfg.entry_block`, `Cfg.exit_block`, `Cfg.reachable_blocks`, `Cfg.preds`, `Cfg.succs`
- Provides: `rpo`, `dominator_tree`, `post_dominator_tree`, `dominance_frontier`, `backedges`, `loop_headers`, `natural_loops`

## Dataflow

### 当前规则

- Dataflow 消费 `LoweredProto + Cfg + GraphFacts`
- Dataflow 产出分析事实，不改写 low-IR
- Dataflow 构建的是 SSA-like 信息，不物化成新的 SSA IR
- open pack 不展开成任意长度寄存器集合，而是单独跟踪
- Dataflow 只做单 proto 内分析
- `Call` 在当前阶段只记录 `EffectTag::Call`
- 不在当前阶段递归分析 callee 的 capture / upvalue / env 副作用

### 数据结构

```text
DataflowFacts
  instr_effects: Vec<InstrEffect>
  effect_summaries: Vec<SideEffectSummary>
  defs
  reg_versions
  instr_defs
  reaching_defs
  use_defs
  def_uses
  live_in
  live_out
  open_reaching_defs
  open_live_in
  open_live_out
  phi_candidates

InstrEffect
  fixed_uses: RegSet
  fixed_must_defs: RegSet
  fixed_may_defs: RegSet
  open_use: Option<Reg>
  open_must_def: Option<Reg>
  open_may_def: Option<Reg>

Def
  id: DefId
  reg: Reg
  instr: InstrRef
  block: BlockRef

DefId
  index

SideEffectSummary
  tags: EffectTagSet

  Alloc
  ReadTable
  WriteTable

  WriteEnv
  ReadUpvalue

PhiCandidate

PhiIncoming

```

### 分析策略

- `phi_candidates` 基于 `dominance_frontier` 生成，并结合真实 use 进行裁剪
- `reaching_defs` 做到 instr 级粒度
- open pack 使用独立的 reaching defs 通道
- open pack 使用独立的 liveness 通道
- `DefId` 是定义实例的唯一身份
- `reg_versions` 记录 `Reg -> DefId` 的版本链关系
- 调试展示可以把 `DefId` 映射成 `rX_n` 形式

### Effect 规则

- `Move` / `LoadNil` / `LoadBool` / `LoadConst` / `UnaryOp` / `BinaryOp` / `Concat` 都只产生固定 `must_def`
- `GetUpvalue` 产生固定 `must_def`，`SetUpvalue` 只产生固定 `use`
- `GetTable` 根据 `base/key` 的寄存器来源产生 `use`，并对 `dst` 产生固定 `must_def`
- `SetTable` / `SetList` 只产生 `use`
- `Call` 的 `callee` 和参数进入 `use`
- `Call.results = Fixed(RegRange)` 产生固定 `must_def`
- `Call.results = Open(Reg)` 产生 `open_must_def`
- `TailCall` 只产生 `use`，不产生 `def`
- `VarArg.results = Fixed(RegRange)` 产生固定 `must_def`
- `VarArg.results = Open(Reg)` 产生 `open_must_def`
- `Return` 只产生 `use`
- `Closure.dst` 产生固定 `must_def`
- `Closure.captures` 中的寄存器来源进入 `use`
- `Closure` 的 upvalue capture 不进入当前 `InstrEffect`
- `NumericForInit` / `NumericForLoop` 对 `index` 产生固定 `must_def`，并消费 `index/limit/step`
- `GenericForCall` 消费 `state`，并按 `results` 产生固定或开放 `must_def`
- `GenericForLoop` 只消费 `control`，不产生 `def`
- `Jump` / `Close` 不产生寄存器 `def/use`
- `Branch` 只消费条件中的寄存器操作数

### Side Effect 规则

- `NewTable` 和 `Closure` 都产生 `Alloc`
- `GetTable(base = Reg)` 产生 `ReadTable`
- `GetTable(base = Env)` 同时产生 `ReadTable` 和 `ReadEnv`
- `SetTable(base = Reg)` 和 `SetList` 产生 `WriteTable`
- `SetTable(base = Env)` 同时产生 `WriteTable` 和 `WriteEnv`
- `GetUpvalue` 产生 `ReadUpvalue`
- `SetUpvalue` 产生 `WriteUpvalue`
- `Call` 只产生 `Call`
- `Close` 产生 `Close`

### Consumes / Provides

- Consumes: `LoweredProto.instrs`, `Cfg`, `GraphFacts`
- Provides: `instr_effects`, `effect_summaries`, `defs`, `reg_versions`, `instr_defs`, `reaching_defs`, `use_defs`, `def_uses`, `phi_candidates`, `live_in`, `live_out`

## 后续依赖

StructureFacts 和后续层会依赖这些事实：

- `GraphFacts.natural_loops` 和 `backedges`
- `GraphFacts.post_dominator_tree`
- `DataflowFacts.phi_candidates`
- `DataflowFacts.use_defs / def_uses`
- `DataflowFacts.effect_summaries`

这些依赖已经足够支撑结构候选提取，不需要在 StructureFacts 层重新回去做底层数据流分析。
