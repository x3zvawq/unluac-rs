# HIR

## 职责

HIR 负责消费 low-IR、图事实、数据流事实与结构候选，做正式恢复决策，并产出 AST 可以直接消费的语义树。

这一层应当：

- 结束“只是候选”的状态
- 选择 branch / loop / short-circuit / unresolved 的正式恢复结果
- 保持变量身份、结构边界和语义保真

这一层不应当：

- 回头重建前层本来该给的事实
- 直接做 AST sugar
- 分配最终字符串命名

## 应消费的前层事实

HIR 应消费：

- `LoweredChunk`
- `CfgGraph`
- `GraphFacts`
- `DataflowFacts`
- `StructureFacts`

HIR 不应直接再把下面这些当成裸存储扫描：

- `dataflow.defs`
- `phi_candidates_in_block`
- `phi.incoming`
- region block 集合

如果确实缺查询或缺结构证据，应优先回到 Dataflow 或 StructureFacts 补事实。

## 应优先复用的共享设施

### 结构恢复共享逻辑

`src/hir/analyze/structure/` 负责结构恢复共享逻辑：

- `body.rs`
- `body/branches.rs`
- `branch_values.rs`
- `loops/`
- `overrides.rs`
- `rewrites.rs`

`entry_overrides`、`phi_overrides`、`suppressed_phis`、`shared expr rewrite` 一类逻辑应继续集中在这里，不要在多个恢复路径平行复制。

### 短路恢复共享逻辑

`src/hir/analyze/short_circuit/` 负责：

- 决策
- guard lowering
- recovery

短路 DAG 的结构语义由 `StructureFacts` 提供，HIR 这里只消费，不再重新发明一套 DAG 事实。

### 表达式 lowering

`src/hir/analyze/exprs/` 已按关注点拆开：

- `regs.rs`
- `access.rs`
- `branch.rs`
- `defs.rs`
- `packs.rs`

表达式恢复问题优先放回这些子模块，不要重新长回单个巨型 `exprs.rs`。

其中像 `CONCAT` 这类“字节码只给了一串操作数、没保留源码括号”的表达式，
HIR 应在 lowering 时就选定稳定 canonical shape。当前约定是优先贴近 Lua 源码默认
语法，例如把连续 `..` canonical 成右结合链，而不是留给 Generate 再机械地在最左边补括号。

### simplify walker / visitor

`src/hir/simplify/walk.rs` 是 HIR simplify pass 的共享 rewrite walker；
`src/hir/simplify/visit.rs` 是 HIR simplify pass 的共享只读 visitor。

新的 simplify pass 应优先接入这两层共享设施，而不是自己复制一套递归骨架。

- 需要“后序遍历后改写 block/stmt/expr”的 pass，优先实现 `HirRewritePass`
- 只需要收集 label/temp/binding 一类事实的 collector，优先实现 `HirVisitor`
- 只有表达式级改写的旧 pass，才继续走 `ExprRewritePass` 兼容层
- 如果 pass 自己要做 block slice rebuild，但只是缺“怎么一致地下钻到所有 child block”，优先复用
  `walk.rs` 里的 nested-block helper，而不是再复制一套 `If/While/Repeat/...` 展开

共享 walker 的遍历契约还包括：

- 所有子节点都必须先完整递归，不能因为前一个子树已经 `changed = true` 就短路跳过后面的子树
- 当前节点自己的 `rewrite_*` hook 也必须总是执行，再把结果和子节点的 `changed` 合并返回

否则像 `if cond then ... end` 这类多子节点节点会出现“条件改了，then/else 就不再继续重写”的
半截遍历，最终让 pass 看起来像是偶发漏改。

如果某个 simplify 文件里出现了手写的 `match HirStmt::{...}` 递归骨架，维护时应先问：
这是不是其实应该回到共享 walker / visitor。

只有当算法本身依赖特定遍历顺序或窗口状态时，才允许保留局部递归 helper，例如：

- 逆序扫描最近一条保留语句
- 同一轮里同时维护“后缀 use-count”和“待删除语句窗口”
- region rebuild 需要把扫描、验证、重写绑在同一个控制流里

这类逻辑的 owner 仍然是该 pass 自己，但“单纯的树递归样板”不应继续散落在各文件里。

## 维护规范

### 1. HIR 决策依赖 StructureFacts

当 HIR 需要知道：

- 哪个 branch arm 合流
- 哪个 loop 有哪些 source-visible binding
- 哪个 short-circuit leaf 对应哪些 value merge

应优先从 StructureFacts 直接读取。

### 2. fallback 只处理“仍然没有结构 owner”的情况

generic unresolved / generic phi materialization 应只处理结构层明确没有接管的剩余部分，不应重新覆盖 branch / loop / short-circuit 已经接管的事实。

### 3. simplify 只做 HIR 内部收敛

HIR simplify 负责：

- temp inline
- locals
- decision
- logical simplify
- table constructors
- close scopes
- carried-local handoff cleanup

它不负责给 StructureFacts 漏掉的事实兜底。

### 4. fixed-point 以阶段为单位维护

`src/hir/simplify.rs` 已把调度写成显式阶段。

新增 pass 时，应先判断它属于：

- `core`
- `exposure`
- `cleanup`

而不是简单把它插进一串 `changed |=` 里。

## 向后提供的事实

HIR 向 AST 正式提供：

- 结构化的语义语句与表达式
- `ParamId` / `LocalId` / `TempId` / `UpvalueId`
- 闭包、表构造器、循环、短路、goto、label 等语义节点
- 对 AST 有意义的最终恢复决策

## 维护检查清单

修改 HIR 时，应至少检查：

1. 这个问题是不是 StructureFacts 没给够证据。
2. 这个 query 是不是应该补到 DataflowFacts / StructureFacts。
3. 这个逻辑是不是应放到 `structure/overrides.rs`、`structure/rewrites.rs` 或 `short_circuit/` 共享目录。
4. 这个改动是不是已经在向 AST sugar 或 Naming 决策越权。
