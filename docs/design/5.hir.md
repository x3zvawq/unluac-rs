# HIR

## 目标

HIR 是结构化后的语义树。

这层已经开始做恢复决策，但还不绑定最终 target dialect 语法，也不负责最终命名和 pretty print。

HIR 的职责是把：

- `Cfg`
- `GraphFacts`
- `DataflowFacts`
- `StructureFacts`

转成更接近源码语义、但仍然保留必要回退能力的中间层。

## 当前规则

- HIR 正式切换到变量世界，不再直接使用 `Reg`
- HIR 允许少量受控 fallback 节点
- HIR 正式支持短路表达式恢复
- HIR 的控制流节点已经接近源码结构
- 最终变量名不在 HIR 阶段确定
- HIR 节点允许挂轻量恢复证据
- `TempId` 允许在 HIR 中继续存在，等待后续折叠清理
- HIR 构造分为“初始恢复”和“HIR simplify”两步

## 变量与命名边界

HIR 使用语义身份，不使用最终打印名。

```text
ParamId
  index

LocalId
  index

UpvalueId
  index

TempId
  index

GlobalRef
  name
```

规则：

- `ParamId`、`LocalId`、`UpvalueId` 是可命名绑定
- `TempId` 是恢复过程中的中间值身份
- `GlobalRef` 表示已经从环境访问恢复出的全局符号引用
- `TempId` 默认不承诺会打印成源码变量
- 表构造器折叠、短路折叠、调用折叠等优化可以直接消除 `TempId`
- 最终命名发生在 HIR 结构稳定之后

这条边界的目的是避免过早按寄存器版本号绑定变量名，导致后续折叠后出现难看的跳号和残留命名。

## 构造策略

HIR 构造分两步：

1. 初始恢复
2. HIR simplify

初始恢复阶段：

- 基于 `StructureFacts`、`DataflowFacts` 和 `GraphFacts` 先构造一个正确但可能保守的 HIR
- 允许保留 `TempId`
- 允许保留少量 fallback 节点
- 不追求最优语法糖或最少中间节点

HIR simplify 阶段：

- 采用 fixed-point pass 迭代收敛
- 只做语义整理和结构稳定化
- 不承担 AST 可读性语法糖职责
- 不把已经恢复出的高层结构再降回低层表示

这里的稳定化指：

- 把已经确认的短路和构造器语义整理成稳定 HIR 节点
- 清理恢复过程中的中间形态
- 不直接做最终 Lua 语法糖选择

当前预期的 HIR simplify pass 包括：

- temp inlining
- short-circuit normalization
- table constructor stabilization
- local declaration merge
- branch simplification
- scope cleanup

## 数据结构

```text

  Number
  String

  TempRef

  Unresolved

HirLValue
  Local(LocalId)
  Upvalue(UpvalueId)
  Global(GlobalRef)
  TableAccess

HirCallExpr
  callee: HirExpr
  args: Vec<HirExpr>
  multiret: bool
  method

HirCallStmt
  call: HirCallExpr

  bindings: Vec<LocalId>
  values: Vec<HirExpr>

HirAssign
  targets: Vec<HirLValue>
  values: Vec<HirExpr>

HirTableConstructor
  array_fields: Vec<HirExpr>
  record_fields: Vec<HirRecordField>
  trailing_multivalue: Option<HirExpr>

HirRecordField
  key: HirTableKey
  value: HirExpr

HirTableKey
  Name
  Expr

HirClosureExpr
  proto: HirProtoRef
  captures
```

规则：

- `Closure` 在 HIR 中是表达式
- HIR 按 proto 拆开保存，不直接 inline 嵌套子 proto
- `HirExpr::Closure` 只引用 `HirProtoRef`
- `CallExpr` 和 `CallStmt` 分开表示
- `Assign` 和 `LocalDecl` 分开表示
- `HirLValue` 单独建模，不复用 `HirExpr`
- `TableConstructor` 在 HIR 中正式建模

## 结构恢复方向

HIR 在这一层做最终恢复决策：

- `ShortCircuitCandidate` 是否恢复成 `LogicalAnd / LogicalOr`
- `BranchCandidate` 是否恢复成 `If`
- `LoopCandidate` 是否恢复成 `While / Repeat / NumericFor / GenericFor`
- `GotoRequirement` 是否强制保留 `Goto / Label`
- `ScopeCandidate` 如何变成嵌套 `Block`

## 恢复证据

HIR 节点允许保留轻量 evidence metadata，用于记录：

- 来源 `BranchCandidate`
- 来源 `LoopCandidate`
- 是否使用了 `ShortCircuitCandidate`
- 是否受 `GotoRequirement` 约束

这层 evidence 只用于调试和恢复策略校验，不重新承载图分析结果。

## Fallback 规则

HIR 允许少量受控 fallback 节点：

- `Goto`
- `Label`
- `Unstructured`
- `Unresolved`

这些节点的作用是：

- 避免在结构不足时过早做错误恢复
- 为后续 AST lowering 提供合法但不完美的保底表示

## Consumes / Provides

- Consumes: `Cfg`, `GraphFacts`, `DataflowFacts`, `StructureFacts`
- Provides: `HirModule`, `HirProto`, `HirStmt`, `HirExpr`, `HirLValue`

## 后续依赖

AST 层会依赖 HIR 提供的这些稳定事实：

- 最终控制结构决策
- 最终表达式恢复决策
- 作用域嵌套关系
- 哪些临时值被保留、哪些被折叠
- 哪些跳转必须以 `goto` 形式保留

也就是说，AST 层不再回头重新做结构恢复，它只负责把 HIR 落到目标 dialect 的语法树。
