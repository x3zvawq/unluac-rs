# HIR

## 目标

HIR 是结构化后的语义树。

这层已经开始做恢复决策，但还不绑定最终 target dialect 语法，也不负责最终命名和 pretty print。

HIR 的职责是把：

- `LoweredChunk`
- `Cfg`
- `GraphFacts`
- `DataflowFacts`
- `StructureFacts`

转成更接近源码语义、但仍然保留必要回退能力的中间层。

## 当前规则

- HIR 正式切换到变量世界，不再直接使用 `Reg`
- HIR 允许少量受控 fallback 节点
- HIR 正式支持短路表达式恢复
- HIR 的控制流节点已经接近源码结构
- 最终变量名不在 HIR 阶段确定
- HIR 节点允许挂轻量恢复证据
- `TempId` 允许在 HIR 中继续存在，等待后续折叠清理
- HIR 构造分为“初始恢复”和“HIR simplify”两步

## 变量与命名边界

HIR 使用语义身份，不使用最终打印名。

```text
ParamId
  index

LocalId
  index

UpvalueId
  index

TempId
  index

GlobalRef
  name
```

规则：

- `ParamId`、`LocalId`、`UpvalueId` 是可命名绑定
- `TempId` 是恢复过程中的中间值身份
- `GlobalRef` 表示已经从环境访问恢复出的全局符号引用
- `TempId` 默认不承诺会打印成源码变量
- 表构造器折叠、短路折叠、调用折叠等优化可以直接消除 `TempId`
- 最终命名发生在 HIR 结构稳定之后
- 第一版 HIR 里，`ParamId / UpvalueId` 直接建模，普通 low-IR `def` 先落成 `TempId`
- `LocalId` 主要在 HIR simplify 和后续作用域整理后逐步提升出来，而不是在初始恢复时激进猜测
- 当前已经实现一条保守的 `TempId -> LocalId` 提升路径：当某个 block 顶层先出现稳定初始化，
  后续又通过简单别名链继续流动，并且跨语句继续被读写时，可以把这一组 temp 合并成同一个
  `LocalId`
- 这条提升会主动避开带 `Goto / Label / Unstructured` 的 block，避免在作用域边界还不稳定时
  过早把 temp 猜成源码变量

这条边界的目的是避免过早按寄存器版本号绑定变量名，导致后续折叠后出现难看的跳号和残留命名。

## 构造策略

HIR 构造分两步：

1. 初始恢复
2. HIR simplify

初始恢复阶段：

- 基于 `LoweredChunk`、`StructureFacts`、`DataflowFacts` 和 `GraphFacts` 先构造一个正确但可能保守的 HIR
- 允许保留 `TempId`
- 允许保留少量 fallback 节点
- 不追求最优语法糖或最少中间节点
- 表达式恢复优先消费 `DataflowFacts.use_values`，而不是直接拿原始 `use_defs` 做多定义猜测
- `ShortCircuitExit::ValueMerge` 会在 merge block 的 phi 物化入口优先恢复成值表达式
- `ShortCircuitCandidate.nodes` 现在是短路 DAG，HIR 会先把它提升成内部共享决策 DAG，再决定能否安全收成 `LogicalAnd / LogicalOr`
- 如果某段 branch 区域已经完全被值短路表达式消费，就不再额外产出重复的 `If`
- 普通 `if/else` 的 merge phi 不再只靠 `PhiCandidate` 裸恢复；当 `StructureFacts` 已经提供
  `branch_value_merge_candidates` 时，HIR 会优先沿这条前层证据恢复：
  能继续视作同一稳定 binding 的，直接保持同一 lvalue 语义；
  需要显式值合流的，先保守落成共享 `Decision`，再在 HIR simplify 末端彻底线性化掉
- 第一轮 loop 恢复已经开始直接消费 `LoopCandidate`：当前只接受 `reducible + 无 goto requirement`
  的基础形态，并先支持 `While / Repeat / NumericFor / GenericFor`
- numeric for 在 HIR 中显式带源码可见的循环绑定，而不是继续沿用 VM 内部 `index/limit/step`
  寄存器世界
- generic for 同样显式带源码可见的 `bindings: Vec<LocalId>`，而不是继续把可见循环变量混在
  VM 内部的 `state/control` 槽位里
- `While / Repeat` 在没有 `goto requirement` 时，已经允许保守恢复一类“统一后继 + cleanup/break pad”
  的多出口循环；这类 pad 会直接在 HIR 中收成 `break`，而不是继续暴露成 `close + jump`
- `GotoRequirement::UnstructuredContinueLike` 不再一律阻止 HIR 结构化；如果它能被证明是当前 active loop
  的 loop-local 控制语义，HIR 可以把它吸收成普通 loop tail，或者恢复成语义级 `Continue`
- loop-local `continue` 也不一定非要先经过 `goto requirement` 暗示：如果某个 branch 的显式边或隐式 merge
  已经直接回到当前 active loop 的 continue target，HIR 也可以把它恢复成 `Continue`
- loop lowering 现在会显式利用 `RegionFact::BranchRegion.blocks` 尊重当前结构 region 的 `stop` 边界，
  避免 loop body 里的 branch 为了追自己的全局 merge 而越界吞掉外层 continue/exit
- `Repeat` 现在允许吸收一种“continue 条件之后再接线性 backedge pad”的 VM 形状，只要这块 pad
  最终只是回到 header，而且它自己的前缀已经被结构化 scope 吞掉
- structured HIR 现在会消费 `scope_candidates.close_points`，把已经被作用域候选吸收的 raw
  `Close` 从结构化路径里抹掉；这些 cleanup 点不再伪装成语句噪音，但后续真正的嵌套 `Block`
  恢复仍然要继续沿着 `ScopeCandidate` 做

HIR simplify 阶段：

- 采用 fixed-point pass 迭代收敛
- fixed-point 允许有保守的轮次上限，避免某条 pass 链震荡时把整个入口卡死
- 只做语义整理和结构稳定化
- 不承担 AST 可读性语法糖职责
- 不把已经恢复出的高层结构再降回低层表示
- 结构性恢复决策优先在初始恢复阶段完成，`simplify` 只负责清理残留中间形态
- `If / Loop / Scope / Goto` 的主恢复不应依赖 `simplify` 兜底，除非只是对已恢复结果做局部收敛
- `LocalId` 提升也属于这种“残留中间形态清理”：只在已经能证明变量身份稳定时，把机械的
  temp 版本号收成源码风格更强的局部绑定
- `Decision` 只允许作为 HIR 内部过渡节点存在；最终 HIR 输出必须把它消掉，重新落成
  普通表达式，或者 `local + if + assign` 这类 AST 可自然消费的结构
- 如果 simplify 达到轮次上限，或者最终 HIR 仍然残留 `Decision / Unresolved / fallback`
  这类中间形态，当前实现可以发出软警告提醒维护者反馈 case；但这类警告不应该被当成
  正常控制流分支长期依赖，更不意味着后层要为它们兜底

这里的稳定化指：

- 把已经确认的短路和构造器语义整理成稳定 HIR 节点
- 清理恢复过程中的中间形态
- 不直接做最终 Lua 语法糖选择

当前预期的 HIR simplify pass 包括：

- temp inlining
- decision DAG canonicalization
- short-circuit normalization
- table constructor stabilization
- conservative local promotion
- local declaration merge
- residual branch cleanup
- residual scope cleanup

其中 `short-circuit normalization` 当前只允许使用在 Lua 值语义下也严格成立的安全恒等式，
例如 `x and (x or y) => x`、`x or (x and y) => x` 这一类；不允许直接照搬布尔代数。
`decision DAG canonicalization` 则负责图层面的通用收敛，例如常量 truthiness 裁剪、
重复 test 消除、基于已知稳定 test truthiness 的 descendant 专门化，以及不可达节点
清理；它仍然是在 HIR 内部解决共享子图问题，而不是通过不断补 case-specific 规则去
“拼回去”。当同一组等价候选既可以写成 guarded disjunction，也可以写成机械因式分解后的
factored conjunction 时，HIR 应优先选择更接近源码短路直觉的 guarded disjunction 形状，
避免把 `or` 两臂的选择关系压成难读的多层 `and`。当前实现已经明确避免“在表达式空间里
做大范围枚举再回填答案”这类高复杂度路径；共享 `Decision DAG` 会优先走确定性的结构提取，
而纯逻辑表达式自然化只做固定模板上的局部等价重写，再交给 simplify fixed-point 多轮收敛。
`conservative local promotion` 当前只覆盖最稳的“顶层初始化 + 简单 temp 别名链 + 后续跨语句
复用”这类模式；它的目标不是一口气识别所有源码局部变量，而是先把最明显的 SSA/temp 噪音
收掉，为后续 AST 和命名阶段提供更自然的绑定形状。当前实现还会处理一类稳定的结构化
`if` 汇合值：如果同一个 temp 在 `if/else` 两边所有会继续 fallthrough 的路径上都完成
赋值，并且分支外还会继续使用，那么 simplify 可以把它提升成 `local x; if ... then x = ...
else x = ... end` 这一类更接近源码的形状。
`repeat ... until` 的条件和 body 共享同一个词法作用域，因此 simplify 在做 temp -> local
提升时，必须把 body 末尾的稳定 local 映射继续带到 `until cond` 上；否则就会得到
“body 里已经是 local，条件里还残留旧 temp”的半截 HIR。
共享 continuation 和共享条件子图的主恢复不再依赖 HIR simplify 的局部规则，而是由
analyze 阶段先保留为决策 DAG；simplify 只负责在这个结果之上做语义严格安全的小幅整理。

```text
  iterator: Vec<HirExpr>
  body: HirBlock

HirGoto
  target

HirLabel
  id

HirUnstructured
  body: HirBlock

```

规则：

- `Closure` 在 HIR 中是表达式
- HIR 按 proto 拆开保存，不直接 inline 嵌套子 proto
- `HirExpr::Closure` 只引用 `HirProtoRef`
- `CallExpr` 和 `CallStmt` 分开表示
- `Assign` 和 `LocalDecl` 分开表示
- `HirLValue` 单独建模，不复用 `HirExpr`
- `HirLValue` 允许 `Temp(TempId)`，因为初始恢复阶段普通寄存器定义默认先落成 temp 绑定
- `HirExpr::Decision` 只用来保留 HIR 内部恢复过程里的共享决策 DAG，避免过早树化时把
  共享子图重复展开；它不是最终 HIR 输出契约，simplify 结束前必须被消掉
- 消费 `ValuePack::Open(start)` 时，要按“固定前缀 + open tail”的连续值包解释；如果真正的 open def 从更晚寄存器开始，HIR 需要先吐出 `start..tail_start-1` 的固定值，再接开放尾值
- `TableSetList` 在 HIR 中是受控语义节点，用来保留 `SETLIST` 的数组段批量写入语义，避免初始恢复阶段又退回字符串化 fallback
- `TableConstructor` 在 HIR 中正式建模，并且通过 `fields: Vec<HirTableField>` 保留数组段和 keyed field 的相对顺序
- 第一版实现允许较多 `Goto / Label / Unstructured / Unresolved` fallback，后续再通过 simplify 收敛
- `Continue` 是 HIR 的语义级循环控制节点，不绑定某个 dialect 的具体语法能力

## 结构恢复方向

HIR 在这一层做最终恢复决策：

- `ShortCircuitCandidate` 是否恢复成 `LogicalAnd / LogicalOr`
- `BranchCandidate` 是否恢复成 `If`
- `BranchValueMergeCandidate` 是否继续沿用同一 binding、还是先恢复成内部 `Decision`
- `LoopCandidate` 是否恢复成 `While / Repeat / NumericFor / GenericFor`
- `GotoRequirement` 是否强制保留 `Goto / Label`
- `ScopeCandidate` 如何变成嵌套 `Block`

其中：

- `ShortCircuitExit::BranchExit` 主要服务条件位置的 `and/or` 恢复
- `ShortCircuitExit::ValueMerge` 主要服务值流位置的短路恢复
- HIR 会先把 `ShortCircuitCandidate` 的 truthy/falsy DAG 提升成共享决策 DAG
- 只有在确认不会因为树化而重复展开共享子图时，HIR 才会把它直接收成 `LogicalAnd / LogicalOr`
- 如果共享 DAG 仍然不能安全收成纯表达式，HIR simplify 需要把它线性化成普通
  `local/if/assign`，而不是把 `Decision` 后移给 AST
- `GotoRequirement::UnstructuredContinueLike` 如果被证明只影响当前 active loop，可以恢复成
  `Continue` 或者直接被吸收到更自然的 `if/break/loop-tail` 结构里
- 其余 `GotoRequirement` 继续保留成 `Goto / Label`，避免 HIR 越权替前层兜底

## 恢复证据

HIR 节点允许保留轻量 evidence metadata，用于记录：

- 来源 `BranchCandidate`
- 来源 `LoopCandidate`
- 是否使用了 `ShortCircuitCandidate`
- 是否受 `GotoRequirement` 约束

这层 evidence 只用于调试和恢复策略校验，不重新承载图分析结果。

## Fallback 规则

HIR 允许少量受控 fallback 节点：

- `Goto`
- `Label`
- `Unstructured`
- `Unresolved`

这些节点的作用是：

- 避免在结构不足时过早做错误恢复
- 为后续 AST lowering 提供合法但不完美的保底表示
- merge 点上的值合流允许先保守落成 `TempId + Unresolved(phi)` 之类的保底形式

## Consumes / Provides

- Consumes: `LoweredChunk`, `Cfg`, `GraphFacts`, `DataflowFacts`, `StructureFacts`
- Provides: `HirModule`, `HirProto`, `HirStmt`, `HirExpr`, `HirLValue`

## 后续依赖

AST 层会依赖 HIR 提供的这些稳定事实：

- 最终控制结构决策
- 最终表达式恢复决策
- 作用域嵌套关系
- 哪些临时值被保留、哪些被折叠
- 哪些跳转必须以 `goto` 形式保留

也就是说，AST 层不再回头重新做结构恢复，它只负责把 HIR 落到目标 dialect 的语法树。
