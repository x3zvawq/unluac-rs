# Readability

## 目标

Readability 负责把已经合法的 AST 收敛成更接近人写源码的 AST。

这一层不重新做结构恢复，不回退到 HIR，不改变 target dialect 的能力边界。

## 当前规则

- Readability 只作用于 AST
- Readability 不回跳到 HIR
- Readability 采用 staged fixed-point
- 每个 pass 必须单调，只能把 AST 推向更稳定、更自然的形式
- Readability 复用前面层提供的副作用和绑定事实，不重新做底层分析
- 调用方可调的源码形状阈值统一定义为共享的 `ReadabilityOptions`
- 这组阈值不仅给 AST Readability 用，前层 HIR 里会改变源码形状的内联也必须直接消费它
- Readability 不能承担“把前层已经过度压扁的表达式重新抽回局部变量”的兜底职责

## 分层

Readability 工作在 `AST lowering` 之后、`Naming` 之前。

顺序固定为：

1. HIR simplify
2. AST lowering
3. AST readability
4. Naming
5. Generate

## 驱动方式

Readability 不是一个完全自由的大循环，而是：

- 按阶段分组 pass
- 每一阶段内固定顺序运行
- 一轮有变化就继续下一轮
- 一轮无变化则结束当前阶段
- 所有阶段完成后进入 Naming

如果后阶段的变化暴露出新的优化机会，只允许在 AST readability 内继续收敛，不允许回到 HIR simplify。

## 当前已落地的 Pass 分组

### 1. Structural Cleanup

- `DeadTempCleanupPass`

目标：

- 删除最终不再需要的中间绑定
- 删除函数体 / chunk 体末尾仅用于落地 VM 结束语义的空 `return`

### 2. Statement Merge

- `AdjacentLocalAssignMergePass`

目标：

- 把 `local a, b` 后面紧跟的 `a, b = ...` 收敛成单条 `local a, b = ...`
- 为后续 `FunctionSugarPass` 和 dump 输出提供更接近源码的声明形状

### 3. Access Sugar

- `FieldAccessSugarPass`

目标：

- 把 `obj["field"]` 这类等价字符串索引规整成 `obj.field`
- 让后续 alias inline / method sugar 统一面对更稳定的前缀表达式形状

### 4. Contextual Expr Inline

- `ContextualExprInlinePass`

目标：

- 只在 AST 自己仍然残留的窄模式上，对 `return`、调用参数、索引位、调用目标做保守内联
- 只内联单次使用、紧邻后一条语句、且可证明安全的表达式
- 允许纯单次 `local` / synthetic local 别名继续折叠，不把它们机械地保留到最终源码里
- 由调用方通过复杂度阈值调节：
  - `return_inline_max_complexity`
  - `index_inline_max_complexity`
  - `args_inline_max_complexity`
  - `access_base_inline_max_complexity`
- 不能作为“前层 HIR 先过度内联、这里再补救”的回收站

这里的 `access_base_inline_max_complexity` 专门控制：

- `obj.field`
- `obj[key]`
- `callee(...)` 里 `callee` 位置的前缀表达式别名折叠

这类访问链里“base 位置”的单段折叠。

它不等同于 `index_inline_max_complexity`：

- `index_inline_max_complexity` 只管 `obj[here]` 里的 `here`
- `access_base_inline_max_complexity` 只管 `here[key]` 里的 `here`

当前实现故意只允许非常窄的一类 access-base 折叠：

- 只折叠到紧邻下一条语句
- 只折叠一段 access chain，不允许连续级联把整条链压扁
- 只允许原子表达式或纯 name-field chain 作为被折叠的 base
- 不把 access-base 折叠继续穿进 `return / 调用参数` 这类位置

### 5. Control-Flow Pretty

- `BranchPolarityPass`

目标：

- 把 `if not cond then A else B end` 收敛成 `if cond then B else A end`
- 让条件分支更接近源码极性

### 6. Function Sugar

- `FunctionSugarPass`
- `LocalFunctionSugarPass`

目标：

- 选择 `function a.b() end`
- 选择 `function a:b() end`
- 选择 `local function f() end`

### 7. Temp Materialize

- `TempMaterializePass`

目标：

- 把最终仍残留在 AST 里的 `TempId` 物化成 AST synthetic locals
- 让 `Naming` 面向“最终局部绑定集合”工作，而不是继续直接面对 HIR temp
- 如果某些 temp 在前面的 pass 里已经消失，这一步不应重新引入任何绑定

## 共享约束

Readability pass 必须共同遵守：

- 不跨越 `Call`、`Close`、`WriteTable`、`WriteEnv`、`WriteUpvalue` 等副作用屏障做危险重排
- 不破坏 `GotoRequirement` 已经固定下来的跳转保留需求
- 不破坏作用域边界
- 不改变 target dialect 不支持的语法
- Readability 自己新增的表达式内联必须受调用方配置约束，不能硬编码成单一路线

这些约束主要复用：

- `DataflowFacts.effect_summaries`
- 绑定 use/def 信息
- 作用域边界
- AST lowering 已经确定的 dialect 约束

## Consumes / Provides

- Consumes: AST, target dialect context, 副作用约束, 绑定约束, `GotoRequirement`
- Provides: 更接近源码、适合进入 Naming 的稳定 AST；该 AST 不应再保留原生 temp 绑定
