# Readability

## 目标

Readability 负责把已经合法的 AST 收敛成更接近人写源码的 AST。

这一层不重新做结构恢复，不回退到 HIR，不改变 target dialect 的能力边界。

## 当前规则

- Readability 只作用于 AST
- Readability 不回跳到 HIR
- Readability 采用 staged fixed-point
- 每个 pass 必须单调，只能把 AST 推向更稳定、更自然的形式
- Readability 复用前面层提供的副作用和绑定事实，不重新做底层分析

## 分层

Readability 工作在 `AST lowering` 之后、`Naming` 之前。

顺序固定为：

1. HIR simplify
2. AST lowering
3. AST readability
4. Naming
5. Generate

## 驱动方式

Readability 不是一个完全自由的大循环，而是：

- 按阶段分组 pass
- 每一阶段内固定顺序运行
- 一轮有变化就继续下一轮
- 一轮无变化则结束当前阶段
- 所有阶段完成后进入 Naming

如果后阶段的变化暴露出新的优化机会，只允许在 AST readability 内继续收敛，不允许回到 HIR simplify。

## 当前 Pass 分组

### 1. Structural Cleanup

- `DoBlockMinimizePass`
- `StatementMergePass`
- `DeadTempCleanupPass`

目标：

- 清理多余 block
- 合并平凡语句
- 删除最终不再需要的中间绑定

### 2. Constructor And Expr Sugar

- `TableConstructorFoldPass`
- `FieldAccessSugarPass`
- `MethodCallSugarPass`
- `ShortCircuitPrettyPass`

目标：

- 选择更自然的表构造器形式
- 把保守的索引访问提升为字段访问
- 把保守的调用形式提升为方法调用
- 让短路表达式更自然

### 3. Function Sugar

- `FunctionSugarPass`
- `LocalFunctionSugarPass`

目标：

- 选择 `function a.b() end`
- 选择 `function a:b() end`
- 选择 `local function f() end`

## 共享约束

Readability pass 必须共同遵守：

- 不跨越 `Call`、`Close`、`WriteTable`、`WriteEnv`、`WriteUpvalue` 等副作用屏障做危险重排
- 不破坏 `GotoRequirement` 已经固定下来的跳转保留需求
- 不破坏作用域边界
- 不改变 target dialect 不支持的语法

这些约束主要复用：

- `DataflowFacts.effect_summaries`
- 绑定 use/def 信息
- 作用域边界
- AST lowering 已经确定的 dialect 约束

## Consumes / Provides

- Consumes: AST, target dialect context, 副作用约束, 绑定约束, `GotoRequirement`
- Provides: 更接近源码、适合进入 Naming 的稳定 AST
