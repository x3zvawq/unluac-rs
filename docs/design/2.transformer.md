# Transformer And Low-IR

## Transformer 角色

Transformer 的职责很窄：

- 消费 raw 层结构
- 完成 dialect-specific lowering
- 产出统一的 low-IR
- 建立 low/raw/debug 之间的映射

Transformer 本身不是重点设计对象，重点是 low-IR。只要 Parser 和 low-IR 定稳，Transformer 的实现基本是按规则转译。

## Lowering 结果

```text
LoweredProto
  instrs: Vec<LowInstr>
  lowering_map: LoweringMap

LoweringMap
  low_to_raw
  raw_to_low
  pc_map
  line_hints
```

规则：

- low-IR 本体不内联 raw pc 和 debug line
- raw instruction、pc、lineinfo 的追踪关系统一放在 `LoweringMap`
- `LoweringMap` 需要支持 `1 -> N` 和 `N -> 1` 的 lowering 关系

## Low-IR 设计方向

- low-IR 使用寄存器机模型
- low-IR 使用纯 statement-based 表达，不构造嵌套表达式
- low-IR 保留少量关键 Lua 语义
- low-IR 不允许 dialect-specific instruction 残留
- 所有 dialect raw instruction 必须在进入 CFG 前完全 lowering 到统一 low-IR

low-IR 的职责是成为统一映射层，服务于：

- CFG
- GraphFacts
- Dataflow
- StructureFacts
- 后续 HIR 恢复

它不直接承担 AST 或源码级表达式职责。

## 已确认的语义重点

- 多返回值
- numeric for
- generic for
- closure
- close 语义
- runtime global-init guard
- variadic proto 的入口 vararg 绑定形态

## 基础类型

```text
Reg
  index

RegRange
  start: Reg
  len

ConstRef
  index

UpvalueRef
  index

ProtoRef
  index

InstrRef
  index

NumberLiteral
  bits

ValueOperand
  Reg(Reg)
  Const(ConstRef)
  Integer(i64)
```

规则：

- 所有基础引用类型都使用 newtype
- 不设计总的 `Ref` 枚举
- 枚举只用于某个字段本身允许多种合法输入的场景
- `ValueOperand` 用于表达 Lua 5.1 等 dialect 中常见的 RK 操作数

## ValueOperand 边界

`ValueOperand = Reg | Const | Integer` 只用于那些在 raw bytecode 里本来就允许寄存器/常量/立即数混合作为输入操作数的位置。

当前明确允许的位置：

- `BinaryOp.lhs`
- `BinaryOp.rhs`
- `SetTable.value`
- `AccessKey`
- `CondOperand`

当前明确不允许扩展成 `ValueOperand` 的位置：

- `Move.src`
- `UnaryOp.src`
- `Concat.src`
- `Call.callee`
- `Call.args`
- `TailCall.callee`
- `TailCall.args`
- `Return.values`
- `VarArg.results`
- `NumericForInit.index/limit/step/binding`
- `NumericForLoop.index/limit/step/binding`
- `GenericForCall.state`
- `GenericForLoop.control`
- `Close.from`
- `GetUpvalue/SetUpvalue`

规则：

- 如果某个 low-IR 槽位在语义上代表“运行时寄存器区间”或“VM 槽位更新”，则保持 `Reg` / `RegRange`
- 不为了减少 `LoadConst` 人为把所有输入都扩成 `ValueOperand`
- 只有 raw 指令本来就存在 RK/寄存器-常量混合输入时，才在 low-IR 中保留这种能力
- `AccessKey` 和 `CondOperand` 虽然当前形状与 `ValueOperand` 相同，但仍保留独立类型名，避免把访问语义、条件语义和值语义混成一个总枚举

## 调用模型

```text
Call
  callee: Reg
  args: ValuePack
  results: ResultPack
  kind: normal | method

TailCall
  callee: Reg
  args: ValuePack
  kind: normal | method

VarArg
  results: ResultPack

Return
  values: ValuePack

ValuePack
  Fixed(RegRange)
  Open(Reg)

ResultPack
  Fixed(RegRange)
  Open(Reg)
  Ignore
```

规则：

- `TailCall` 是独立指令，不复用 `Call + tail = true`
- `Call.kind` 区分普通调用和方法调用
- `Open(Reg)` 表示从某个寄存器开始、一直延续到当前 top 的连续值包
- `Open(Reg)` 允许“固定前缀 + open tail”共存：例如 `return a, f()` 可以 lower 成 `Return values = Open(r_a)`，其中 `r_a..r_f-1` 仍然是固定值，真正的开放尾部从 `f()` 的结果起点开始
- 后续层消费 `Open(Reg)` 时，必须按 `start..tail_start-1` 的固定前缀 + open tail 解释，不能把整个 pack 简化成单个“开放值”
- `Ignore` 表示调用结果被直接丢弃
- `Open(Reg)` 只用于合法的多值传播场景

## 访问模型

```text
GetTable
  dst: Reg
  base: AccessBase
  key: AccessKey

SetTable
  base: AccessBase
  key: AccessKey
  value: ValueOperand

NewTable
  dst: Reg

SetList
  base: Reg
  values: ValuePack
  start_index

AccessBase
  Reg(Reg)
  Env
  Upvalue(UpvalueRef)

AccessKey
  Reg(Reg)
  Const(ConstRef)
  Integer(i64)
```

规则：

- 不单独设计 `GetGlobal/SetGlobal`
- global access 统一 lower 成 `GetTable/SetTable(base = Env, key = Const)`
- 对于 `GETTABUP/SETTABUP` 这类“以 upvalue table 为 base”的访问，必须保留真实 `AccessBase::Upvalue`，不能伪装成 `_ENV`
- 显式 `_G` / `_ENV` 不在 low-IR 特殊化
- `GetUpvalue/SetUpvalue` 保持独立，不并入 table access 模型
- `SetList.values` 允许使用 `Open(Reg)`
- `SetTable.value` 允许直接使用 `Const(ConstRef)`，避免为 RK value 人工插入无意义 `LoadConst`
- `AccessKey::Integer` 用于 Lua 5.4/5.5 这类会把小整数 key 直接编码在 raw 指令里的版本

## 方言特定 lowering 契约

- `ERRNNIL` 必须 lower 成显式 `LowInstr::ErrNil`，不能在 Transformer 里静默抹掉
- `TBC` 必须 lower 成显式 `LowInstr::Tbc`，不能退回字符串化 cleanup 注释
- Lua 5.5 的 `GETVARG` 不新增专属 low-IR 指令；它应 lower 成普通 `GetTable`，`base`
  指向 variadic proto 的入口 vararg 参数寄存器
- 如果 parser 已经把 `EXTRAARG` 折叠进 raw 指令，Transformer 就不再接收游离 `ExtraArg`
  形态；这一层只消费已经补全过扩展信息的 raw 指令
- `VarArgPrep`、`MMBIN*` 这类 VM 辅助指令只有在其语义效果已经由其他 low-IR 明确保留时，
  才允许被吸收

## 闭包模型

```text
Closure
  dst: Reg
  proto: ProtoRef
  captures: Vec<Capture>

Capture
  source: CaptureSource
  extra: DialectCaptureExtra

CaptureSource
  Reg(Reg)
  Upvalue(UpvalueRef)
```

规则：

- `Closure` 把闭包创建和 capture 初始化统一成单条指令
- `captures` 显式进入 low-IR
- capture 采用 `common + extra` 思路
- `Closure` 不承担函数命名或源码级定义形式信息

## 循环模型

```text
NumericForInit
  index: Reg
  limit: Reg
  step: Reg
  binding: Reg
  body_target: InstrRef
  exit_target: InstrRef

NumericForLoop
  index: Reg
  limit: Reg
  step: Reg
  binding: Reg
  body_target: InstrRef
  exit_target: InstrRef

GenericForCall
  state: RegRange
  results: ResultPack

GenericForLoop
  control: Reg
  bindings: RegRange
  body_target: InstrRef
  exit_target: InstrRef
```

规则：

- `NumericForInit`、`NumericForLoop`、`GenericForLoop` 都是结构化 terminator
- generic for 保持 `call + loop` 两条指令
- 普通条件跳转使用 `Branch`
- loop 控制跳转由专门 loop terminator 表达
- `NumericForInit.binding`、`NumericForLoop.binding` 显式保留 Lua 可见的循环变量寄存器
- `GenericForLoop.bindings` 显式保留 Lua 可见的循环变量寄存器区间
- `GenericForLoop.control` 只表示 VM 内部的 control 槽位，不再把可见 bindings 混进同一个 `RegRange`
- 这样后续层不需要再反推某个 dialect 的 VM 布局细节，就能稳定恢复 `for i = ...`
- 同样地，后续层也不需要再猜 generic-for 的 `A+2 / A+3..` 约定，就能稳定恢复 `for k, v in ...`

## 控制流模型

```text
Jump
  target: InstrRef

Branch
  cond: BranchCond
  then_target: InstrRef
  else_target: InstrRef

BranchCond
  predicate: BranchPredicate
  operands: BranchOperands
  negated: bool

BranchPredicate
  Truthy
  Eq
  Lt
  Le

BranchOperands
  Unary(CondOperand)
  Binary(CondOperand, CondOperand)

CondOperand
  Reg(Reg)
  Const(ConstRef)
```

规则：

- `Jump` 表示无条件跳转
- `Branch` 表示条件跳转
- 线性 low-IR 阶段使用 `InstrRef`
- `BranchCond` 必须是无副作用条件
- `TestSet` 一类模式在 transformer 阶段拆解，不进入 low-IR
- `nil` 判断统一使用 `Eq(..., nil)`，不单独设计 `IsNil`
- `BranchCond` 允许比较寄存器与常量，不要求 transformer 先人为插入 `LoadConst`

## 合法性约束

- low-IR 是单个 proto 内的线性指令序列
- 所有 `InstrRef` 都必须指向当前 proto 内合法指令
- low-IR 中不允许出现 `Unknown`、`Nop` 或方言残留指令
- `Closure` 必须已经完成 capture lowering
- `BranchCond` 必须无副作用
- terminator 包括 `Jump`、`Branch`、`TailCall`、`Return`、`NumericForInit`、`NumericForLoop`、`GenericForLoop`
- 非 terminator 默认顺序流向下一条指令
- terminator 后允许继续存在后续指令，CFG 再判断可达性
- unreachable 指令允许保留在 low-IR 中
- `Open(Reg)` 只允许出现在约定的多值传播位置
- `ConstRef`、`UpvalueRef`、`ProtoRef` 都必须在当前 proto 上下文中合法
- raw 的控制流模式、闭包模式、环境访问模式和开放值包语义都必须在进入 CFG 前完成 lowering
- runtime cleanup、global-init guard、以及 variadic entry binding 这些会影响后续结构恢复的语义，
  也必须在进入 CFG 前完成稳定 lowering

## Consumes / Provides

- Consumes: `RawChunk`, `RawProto`, `RawInstr`, `ChunkHeader`, raw 常量/upvalue/debug 信息
- Provides: `LoweredProto`, `LowInstr`, `LoweringMap`

## 后续依赖

CFG 和后续层会依赖 low-IR 的这些事实：

- terminator 集合稳定
- 跳转 target 全部合法
- branch 条件没有副作用
- loop terminator 自带完整控制流目标
- closure/capture 已 fully lowered
- `Open(Reg)` 只出现在合法位置
- `ErrNil` / `Tbc` / `Close` 这类语义级节点不会在前端被静默丢弃
- 没有方言残留指令

因此这些约束必须在 Transformer 阶段一次性满足。
