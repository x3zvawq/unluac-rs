# Naming

## 目标

Naming 负责把 AST 中的绑定引用和符号引用转换成最终字符串标识符。

这层发生在：

- HIR 恢复稳定之后
- AST 已经合法语法化之后
- Readability 已经完成之后
- Generate 之前

## 当前规则

- Naming 不参与结构恢复
- Naming 不再删除或新增语法结构
- Naming 面向最终保留下来的绑定集合工作
- Naming 的输入必须是稳定 AST
- Naming 默认不改写 AST 的绑定身份，而是产出独立 NameMap

## 命名对象

Naming 处理：

- `ParamId`
- `LocalId`
- `UpvalueId`
- 仍被物化保留下来的 `TempId`

全局引用不通过 Naming 发明名字，而是直接使用 AST lowering 已经确定的全局名字或名字路径。

## Upvalue 规则

- `UpvalueId` 有独立身份，不与外层 `LocalId/ParamId/UpvalueId` 复用 id
- `UpvalueId` 可以记录它捕获自哪个外层绑定
- 默认情况下，`UpvalueId` 继承被捕获绑定的最终名字
- 如果当前作用域发生冲突，再在冲突消解阶段改名

## 命名模式

```text
NamingMode
  DebugLike
  Simple
  Heuristic
```

### DebugLike

使用接近 SSA/反编译器风格的命名：

- `r{proto}_{reg}_{version}`
- 或 `r{reg}_{version}`

可配置是否带 proto 编号。

这个模式的特点是：

- 最贴近底层来源
- 便于从结果反推字节码布局
- 可读性相对一般，但调试价值最高

### Simple

使用简单递增命名：

- `a, b, c`
- `x, y, z`
- `i, j, k`

这个模式的特点是：

- 输出干净
- 推断少
- 风格稳定

### Heuristic

使用带一定推断的源码风格命名：

- 表构造器绑定：`t`, `tbl`, `map`
- 返回值：`ret`, `result`
- 参数：`arg1`, `arg2`
- 迭代变量：`i`, `k`, `v`
- 布尔值：`ok`, `flag`

这个模式最接近人写源码，但也依赖更多结构推断。

## 名字来源优先级

最终命名按如下优先级决策：

1. debug info 中可用且可信的名字
2. 由 `NamingMode` 生成的候选名
3. 冲突消解后的稳定 fallback 名

## 输出形式

```text
NameMap
  param_names
  local_names
  upvalue_names
  temp_names
```

规则：

- AST 继续保留 `ParamId / LocalId / UpvalueId / TempId` 等绑定引用
- Naming 产出独立的 `NameMap`
- Generate 通过 `AST + NameMap` 输出最终文本
- 如果需要调试或测试，仍然可以同时观察绑定身份和最终字符串

## self 规则

如果前面阶段已经确定方法语法糖成立：

- `function obj:add(...)`
- `obj:add(...)`

那么对应函数的接收者参数优先命名为 `self`。

## 冲突消解

Naming 必须统一处理：

- 同作用域重名
- 外层遮蔽
- Lua 关键字冲突
- 与已有 label / function name / parameter 的冲突

冲突消解发生在所有候选名选定之后，不应在更早阶段零散处理。

## Consumes / Provides

- Consumes: 稳定 AST、绑定引用、作用域信息、debug info 中可复用的名字、readability 之后最终保留的绑定集合
- Provides: `NameMap`
