# Naming

## 目标

Naming 负责把 AST 中的绑定引用和符号引用转换成最终字符串标识符。

这层发生在：

- HIR 恢复稳定之后
- AST 已经合法语法化之后
- Readability 已经完成之后
- Generate 之前

## 当前规则

- Naming 不参与结构恢复
- Naming 不再删除或新增语法结构
- Naming 面向最终保留下来的绑定集合工作
- Naming 的输入必须是稳定 AST
- Naming 默认不改写 AST 的绑定身份，而是产出独立 NameMap
- Naming 必须在 stripped chunk 上稳定工作
- debug info 只能作为辅助证据，不能成为 Naming 正确性的必要前提
- Naming 自己负责从 AST 推导 lexical scope，不假定前层已经额外导出一份独立 scope facts
- 如果 Readability 之后仍残留原生 `TempId`，应视为前层边界没有收敛好，而不是让 Naming 继续直接消费它

## 命名对象

Naming 处理：

- `ParamId`
- `LocalId`
- `UpvalueId`
- Readability 末尾物化出来的 AST synthetic locals

全局引用不通过 Naming 发明名字，而是直接使用 AST lowering 已经确定的全局名字或名字路径。

## Upvalue 规则

- `UpvalueId` 有独立身份，不与外层 `LocalId/ParamId/UpvalueId` 复用 id
- 如果前层保留了足够的 capture provenance，`UpvalueId` 可以记录它捕获自哪个外层绑定
- 只有在 capture provenance 或其它证据足够时，`UpvalueId` 才应继承被捕获绑定的最终名字
- 如果当前作用域发生冲突，再在冲突消解阶段改名
- `DebugLike` 也应优先继承 capture provenance；只有拿不到父绑定来源时，才退回稳定 `u*` 形式

## 命名模式

```text
NamingMode
  DebugLike
  Simple
  Heuristic
```

### DebugLike

使用接近稳定 binding id / 反编译器风格的命名：

- `p{proto}_{index}`
- `r{proto}_{index}`
- `u{proto}_{index}`

其中：

- 普通 `LocalId` 使用 `r`
- Readability 物化出来的 synthetic local 也使用 `r`
- synthetic local 的编号接在显式 local 之后，形成同一函数内统一的局部槽位视角
- `UpvalueId` 如果能追溯到父绑定，则直接沿用父绑定的最终名字；只有证据不足时才使用 `u`

也可以配置为省略 `proto` 前缀，只保留函数内局部编号。

这个模式的特点是：

- 最贴近底层来源
- 便于从结果反推 binding 身份
- 可读性相对一般，但调试价值最高
- 为了保持“稳定槽位视角”，该模式不依赖语义 hint，也不优先采用 debug info 里的源码名

### Simple

使用简单递增命名：

- `a, b, c`
- `x, y, z`
- `i, j, k`

这个模式的特点是：

- 输出干净
- 推断少
- 风格稳定
- 为了避免嵌套函数全部重名成 `fn`，`FunctionShape` 候选会做模块级递增，例如 `fn`、`fn2`、`fn3`
- 对 stripped chunk 里自动生成的参数名，Naming 会避开“函数定义点当前可见的外层绑定名”
  例如外层已经自动拿到 `a, b` 时，里层参数会优先继续取 `c, d`，而不是机械重用 `a, b`

### Heuristic

使用带一定推断的源码风格命名：

- 表构造器绑定：`t`, `tbl`, `map`
- 返回值：`ret`, `result`
- 参数：`arg1`, `arg2`
- 迭代变量：`i`, `k`, `v`
- 布尔值：`ok`, `flag`

这个模式最接近人写源码，但也依赖更多结构推断。

同时它沿用和 `Simple` 一样的函数形状去重规则：

- `FunctionShape` 候选会做模块级递增，避免每层函数都重新从 `fn` 开始
- 自动生成的参数名同样会避开函数定义点当前可见的外层绑定名

## 名字来源优先级

最终命名按如下优先级决策：

1. debug info 中可用且可信的名字
2. 来自 AST 结构的稳定候选名
3. 由 `NamingMode` 生成的候选名
4. 冲突消解后的稳定 fallback 名

例外：

- `DebugLike` 模式对 `ParamId / LocalId / synthetic local` 刻意跳过前两类语义来源，直接使用稳定槽位命名
- `DebugLike` 的 `UpvalueId` 是特例：它优先继承 capture provenance，因为 upvalue 本质上是父作用域绑定的延续，而不是当前函数新发明的槽位名

这里的 AST 结构候选至少包括：

- numeric-for / generic-for 的循环角色
- function / method 语法糖已经恢复出的结构
- table constructor、field access、boolean shell 等相对稳定的语义形状

没有任何证据时，Naming 仍必须产出稳定名字，而不是失败或退回依赖 debug info。

## 作用域来源

Naming 不假定前层已经单独提供 `ScopeFacts`。

第一版 Naming 的作用域来源应当是：

- `AstModule` / `AstBlock` / `AstStmt`
- 函数嵌套边界
- 局部声明、循环绑定和参数列表

也就是说，Naming 需要通过 AST 自己推导：

- 某个 binding 从哪里开始可见
- 哪些 binding 处在同一词法可见域
- 哪些名字冲突是必须避免的

在需要保守处理时，允许先采用“同一函数内全局唯一”这一更严格约束，
但不能反过来假定没有作用域信息就随意重名。

## 命名证据

Naming 除了稳定 AST 之外，还可以消费一份可选的 `NamingEvidence`。

```text
NamingEvidence
  functions: Vec<FunctionNamingEvidence>

FunctionNamingEvidence
  param_debug_names
  local_debug_names
  upvalue_debug_names
  upvalue_capture_sources
  temp_debug_names
  child_functions
```

规则：

- 这是“辅助证据”，不是 Naming 的必需输入
- parser / HIR 可以提供其中一部分，也可以只提供很有限的子集
- 没有证据的槽位用 `None`
- stripped chunk 下允许整份证据基本为空
- Naming 不能因为 `NamingEvidence` 不完整就失去稳定性

当前阶段最现实的证据来源包括：

- parser 的 `local_vars`
- parser 的 `upvalue_names`
- HIR closure capture 里“child upvalue 捕获自 parent 哪个 binding”的 provenance
- HIR 的 `temp_debug_locals`

但这些证据目前并不保证覆盖所有 `ParamId / LocalId / UpvalueId / synthetic local`。

## 输出形式

```text
NameMap
  entry_function
  functions: Vec<FunctionNameMap>

FunctionNameMap
  params
  locals
  synthetic_locals
  upvalues

NameInfo
  text
  source
  renamed
```

规则：

- AST 在进入 Naming 时继续保留 `ParamId / LocalId / UpvalueId / synthetic local` 等绑定引用
- Naming 产出独立的 `NameMap`
- Generate 通过 `AST + NameMap` 输出最终文本
- 如果需要调试或测试，仍然可以同时观察绑定身份和最终字符串
- 由于 `ParamId / LocalId / UpvalueId / synthetic local` 的编号只在单个函数上下文内稳定，`NameMap` 不能被建模成跨整个模块的单一平铺 map
- 更合理的建模是“按函数上下文组织的名字集合”，并且该上下文需要和 AST 的稳定函数身份对齐

## self 规则

如果前面阶段已经显式确定方法语法糖成立：

- `function obj:add(...)`
- `obj:add(...)`

那么对应函数的接收者参数优先命名为 `self`。

如果前面阶段没有显式产出 method declaration 证据，Naming 不能仅凭“看起来像方法调用”就擅自发明 `self`。

## 冲突消解

Naming 必须统一处理：

- 同作用域重名
- 外层遮蔽
- Lua 关键字冲突
- 与已有 label / function name / parameter 的冲突

冲突消解发生在所有候选名选定之后，不应在更早阶段零散处理。

## Consumes / Provides

- Consumes: 稳定 AST、稳定函数身份、绑定引用、readability 之后最终保留的绑定集合、可选 `NamingEvidence`
- Provides: `NameMap`
