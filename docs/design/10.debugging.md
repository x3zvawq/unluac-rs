# Debugging 工具

> 本文记录仓库提供的所有调试入口及其使用方式，是排查问题、定位回归的第一阅读文档。

## 调试命令速查

```sh
# 查看 parse 阶段产物
unluac --dialect lua54 --debug --target-stage parse <file.luac>

# 依次查看各层
--target-stage transform       # low-IR
--target-stage cfg             # CFG basic blocks
--target-stage graph-facts     # 支配树 / 回边 / 自然循环
--target-stage dataflow        # def / phi / liveness
--target-stage structure-facts # 结构候选
--target-stage hir             # HIR 结构
--target-stage readability     # Readability 整形后的 AST
--target-stage naming          # NameMap
--target-stage generate        # 生成结果

# 查看某一 simplify pass 的 before / after
unluac --dump-pass decision <file.luac>
unluac --dump-pass decision,temp-inline <file.luac>   # 多个 pass 用逗号分隔
unluac --dump-pass decision --dump-pass-proto 1 <file.luac>   # 只转储子 proto #1
```

## CLI 参数速查（调试相关）

| 参数 | 类型 | 说明 |
|---|---|---|
| `--debug` | flag | 开启详细调试输出 |
| `--target-stage <stage>` | string | 打印到指定阶段后停止 |
| `--dump-pass <name[,name...]>` | string（可重复） | 打印特定 pass 的 before/after diff |
| `--dump-pass-proto <index>` | u32（可重复） | 限制 dump 仅输出子 proto（0=主 proto，1=第一个子 proto…） |

## `--dump-pass` 工作方式

1. CLI 解析出 `--dump-pass name1,name2` → `Vec<String>`，传入 `DecompileOptions::dump_pass`。
2. Pipeline 在 `resolve_output_plan` 里构建 `PassDumpConfig { enabled_passes, proto_filter }`。
3. HIR simplify 和 AST Readability 两个 invalidation loop 各自在每个 pass 前后：
   - 若 pass 名命中 `PassDumpConfig::enabled_passes`，且当前 proto index 在 `proto_filter`（或 filter 为空）：
   - 打印 `=== before: <pass-name> ===` + 当前快照
   - 执行 pass
   - 打印 `=== after: <pass-name> ===` + 更新后快照
4. 快照格式由 `src/hir/debug.rs::dump_proto_snapshot` / `src/ast/debug.rs::dump_ast_snapshot` 负责。

### `PassDumpConfig` 位置

```
src/hir/mod.rs::PassDumpConfig   (pub(crate))
  enabled_passes: HashSet<String>
  proto_filter: Option<Vec<u32>>
```

## Timing / Profiling

`src/timing.rs` 提供 `Timings` 结构，pipeline 内通过 `timings.record("stage", duration)` 打点。
在 debug 模式下 `decompile/debug.rs` 会打印各阶段耗时。

## Debug 颜色输出

`src/debug/colorize.rs` 提供 ANSI 着色辅助。默认在终端输出时激活，文件重定向时关闭。

## 调试 Lua cases

- `tests/lua_cases/` — 持久测试样例，commit 后长期有效
- `tmp/` — 临时调试文件（可随时写，不保证持久）

## 排错流程

1. 确认错误阶段：逐 stage 打印，从 parse 开始往后推。
2. 找到第一层结果不正确的阶段 → 阅读该层的 `design/X.yyy.md`。
3. 尽可能把根因追溯到最早的层（"是 StructureFacts 漏了候选？还是 HIR lowering 没消费？"）。
4. 不在后层特判或兜底，在根因层修复。
5. 修改后补 `tests/lua_cases/` 测试样例防回归。

## 新增 pass 时的调试设施核对清单

- [ ] `PASS_DESCRIPTORS` 中 `name` 字符串拼写是否和 `--dump-pass` 期望的相同？
- [ ] `dump_proto_snapshot` / `dump_ast_snapshot` 是否覆盖新节点类型？
- [ ] `PassDumpConfig::enabled_passes` 中的 pass 名是精确字符串匹配（非前缀）。
