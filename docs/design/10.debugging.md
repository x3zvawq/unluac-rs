# Debugging

## 目标

调试工具链负责为每一层提供稳定、可读、低噪音的检查输出。

它的目标不是替代 Rust 的 `Debug`，而是提供一套面向开发和排错的专用视图，避免：

- 结构体递归打印过长
- 不同层之间缺少稳定对照
- 输出过于冗杂，难以快速定位问题
- AI 或人类在阅读调试输出时被低价值字段淹没

## 总体原则

- 每一层都有自己的专用 dump 视图，不直接复用 `#[derive(Debug)]`
- 所有 dump 都使用稳定 id，而不是依赖内存地址或遍历偶然顺序
- 所有 dump 都支持按 proto、block、instr、reg、def 等维度过滤
- 所有 dump 都尽量一屏看懂，详细信息通过 `--verbose` 或二级视图展开
- 所有跨层调试都通过映射关系串联，而不是人工反查
- 每完成一层正式实现，就同步完成该层的 dump 能力

## 调试对象

建议覆盖这些层次：

1. parser
2. transformer / low-IR
3. cfg
4. graph-facts
5. dataflow
6. structure
7. hir
8. ast
9. naming
10. generate

## 调试输出形态

面向人类阅读，默认使用：

- 紧凑文本
- 缩进清晰
- 稳定 id
- 只显示关键字段

这是日常开发时最常用的视图。

推荐风格：

- 终端友好的 line-oriented dump
- 先输出层级 header，再输出 block，再输出逐指令行
- 每条指令一行，尽量把关键信息压到同一行
- 默认优先显示“当前排错最值钱的信息”，而不是完整字段转储
- 逐指令行尽量遵守统一列顺序，方便人眼扫描和文本 diff

推荐列顺序：

1. `inst id`
2. `label`
3. `opcode summary`
4. `origin`
5. `reads`
6. `writes`
7. `effects`
8. `notes`

规则：

- `effects` 一列始终显示
- 没有效果标签时也输出 `effects=[-]`

## 集成方式

调试能力优先集成进主反编译 pipeline，而不是先设计成独立命令系统。

建议提供：

```text
DebugOptions
  enable
  output_stage
  stop_after_stage
  format
  detail
  filters
```

典型字段：

- `enable: bool`
- `output_stage`: 指定输出哪一层的 dump
- `stop_after_stage`: 在某层结束后直接停止 pipeline
- `format`: `human | json`
- `detail`: `summary | normal | verbose`
- `filters`: proto/block/instr/def/reg/range 等过滤器

主 pipeline 根据 `DebugOptions`：

- 在对应层完成后调用该层的 dump 方法
- 如设置了 `stop_after_stage`，则在该层结束后提前返回

## API 方向

调试工具优先以方法形式暴露，而不是强绑定为独立 CLI 子命令。

例如每层都可提供等价能力：

- `dump_parser(...)`
- `dump_lir(...)`
- `dump_cfg(...)`
- `dump_graph_facts(...)`
- `dump_dataflow(...)`
- `dump_structure(...)`
- `dump_hir(...)`
- `dump_ast(...)`
- `dump_naming(...)`
- `dump_generate(...)`

主反编译入口只负责根据 `DebugOptions` 选择是否调用它们。

这样做的好处是：

- library 和 CLI 共享同一套调试能力
- 单测、基准测试和开发时都能直接复用
- 不要求一开始就把调试系统设计成一堆独立命令

## 统一过滤器

所有调试子命令应共享一套过滤器：

```text
--proto <id>
--block <id>
--instr <id>
--def <id>
--reg <id>
--range <start:end>
--reachable-only
--verbose
--format human|json
```

规则：

- 默认只输出当前最相关范围
- `--verbose` 才展开次要字段
- `--reachable-only` 默认开启，必要时可关闭查看完整图

## 各层推荐视图

## Parser

提供：

- `header` 视图
- `proto tree` 视图
- `constants` 视图
- `raw instructions` 视图

Human View 示例目标：

- header 一屏看完
- proto 树按缩进显示
- 常量表按 index 显示
- 指令显示 `pc / opcode / operands / origin`

重点是能快速回答：

- 解析有没有丢字段
- 指令 decode 是否正确
- 某个 proto 的常量和子 proto 是否对齐

## Transformer / Low-IR

提供：

- `low-ir listing`
- `lowering-map`
- `raw -> low`
- `low -> raw`

Human View 示例目标：

- 每条 low-instr 一行
- 显示 `instr id / opcode-like summary / defs / uses`
- 可选显示它对应哪些 raw pc

推荐格式：

```text
===== Dump LIR =====
lir lua5.1 entry=proto#0 functions=2

fn proto#0 params=0+ upvals=0 blocks=1
  inputs: ...=v0
  closures:
    #000 block=#0 r0 <- proto#1 captures=[]

  block #0 preds=- succs=- sealed=true
    versions: r0=v1, r1=v2, ...
    inst=@0  label=L0  closure   r0 <- proto#1 captures=[] origin=[pc=0 opcode=CLOSURE] ; reads=[-] writes=[dst=r0:v1] effects=[alloc]
    inst=@1  label=L1  get-table r1 <- _ENV[k0="print"] origin=[pc=1 opcode=GETGLOBAL] ; reads=[-] writes=[dst=r1:v2] effects=[rmem, rglob]
    inst=@2  label=L2  load-const r2 <- k1="branch-state" origin=[pc=2 opcode=LOADK] ; reads=[-] writes=[dst=r2:v3] effects=[-]
```

这类格式的特点是：

- proto header 一眼能看出函数边界
- block header 一眼能看出图结构
- `versions / reads / writes / effects / origin` 同行出现，方便横向比对
- 适合作为日常主视图

重点是能快速回答：

- 某条 raw 指令被 lower 成了什么
- 哪些模式被合并成一条 low-IR
- 哪里发生了 `1 -> N` 或 `N -> 1`

## CFG

提供：

- block listing
- edge listing
- cfg summary

Human View 示例目标：

- block 显示 `block id / instr range / preds / succs`
- edge 单独列出 `from -> to (kind)`
- 可选打印每个 block 末尾 terminator

重点是能快速回答：

- 切块是否正确
- edge kind 是否正确
- synthetic exit 是否正确接边

## GraphFacts

提供：

- dominator tree
- post-dominator tree
- dominance frontier
- backedges
- natural loops

Human View 示例目标：

- dominator tree 用树形或 parent 列表显示
- natural loop 列出 `header / backedge / blocks`
- frontier 用紧凑 block id 集合显示

重点是能快速回答：

- 某 block 的 idom/ipdom 是谁
- 某 backedge 是否被正确识别
- 某 loop 候选覆盖了哪些 block

## Dataflow

提供：

- instr effects
- side effects
- reaching defs
- use-def / def-use
- phi candidates
- liveness

Human View 示例目标：

- 每条指令显示 `uses / defs / open-use / open-def`
- `def_17 (r0) -> uses[...]`
- `instr 42 lhs-use -> defs[...]`
- `phi block B7 reg r0 <- (B3:def_12, B5:def_18)`

推荐风格：

- block 内先打印当前版本表
- 再按指令打印 `reads / writes / effects`
- `DefId`、`UseId`、`version`、`origin` 尽量同行
- `phi_candidates` 单独列在 block 尾或函数尾，避免打断主列表

重点是能快速回答：

- 版本链是否正确
- 某 use 到底接到了哪些 def
- open pack 有没有被错误当成固定寄存器

## StructureFacts

提供：

- branch candidates
- loop candidates
- short-circuit candidates
- goto requirements
- scope candidates

Human View 示例目标：

- 候选按 header block 聚合显示
- `kind_hint / merge / exits / reducible`
- `goto requirement` 单独列出原因

重点是能快速回答：

- 为什么这里被识别成 loop/branch/short-circuit
- 为什么这里必须保留 goto

## HIR

提供：

- stmt tree
- expr tree
- binding usage
- fallback nodes

Human View 示例目标：

- 语句树按 block/缩进显示
- 表达式树紧凑显示
- 单独标记 `TempId` 和 fallback 节点

重点是能快速回答：

- HIR 恢复决策是否正确
- 哪些 temp 还没有消掉
- 哪些地方还保留了 `Goto` 或 `Unresolved`

## AST

提供：

- target-aware syntax tree
- sugar decisions

Human View 示例目标：

- 显示 `FieldAccess / IndexAccess / MethodCall / FunctionDecl`
- 标出某些 sugar 是从什么保守形式提升来的

重点是能快速回答：

- target dialect 约束有没有生效
- 为什么是 `obj.x` 而不是 `obj["x"]`
- 为什么是 `function a:b()` 而不是赋值

## Naming

提供：

- binding -> final name
- name source
- conflict resolution

Human View 示例目标：

- `LocalId(3) -> value (source=debug)`
- `TempId(8) -> t2 (source=heuristic, renamed-for-conflict)`

重点是能快速回答：

- 某个名字从哪里来
- 为什么被改名
- upvalue 为什么沿用了外层名字

## Generate

提供：

- final source
- optional doc tree
- line-break decisions

Human View 示例目标：

- 默认看最终源码
- `--verbose` 时可显示 doc tree 或布局决策摘要

重点是能快速回答：

- 为什么这里换行
- 为什么表构造器展开
- 某个 target option 如何影响输出

## 跨层对照

调试工具链必须能支持跨层跳转：

- raw instr -> low instr
- low instr -> block
- block -> loop candidate / branch candidate
- low instr -> def/use
- HIR stmt/expr -> source low instr evidence
- AST 节点 -> HIR evidence
- binding id -> final name

这要求各层都保留稳定 id 和必要映射。

## 输出分级

建议所有 human dump 至少支持三档：

### 1. summary

- 只显示 proto / block 级摘要
- 不展开逐指令

### 2. normal

- 默认档
- 显示逐指令单行摘要
- 显示 `origin / reads / writes / effects`

### 3. verbose

- 展开额外映射、来源、候选、evidence
- 只在具体排错时使用

这样可以避免默认输出过大、过吵、过烧 token。

## 命令建议

如果后续需要 CLI，再统一到一个命令入口，例如：

```text
unluac debug parser ...
unluac debug lir ...
unluac debug cfg ...
unluac debug graph ...
unluac debug dataflow ...
unluac debug structure ...
unluac debug hir ...
unluac debug ast ...
unluac debug naming ...
unluac debug generate ...
```

这样好处是：

- 工具习惯统一
- 过滤器统一
- 输出风格统一

但这不是当前设计的前置要求。

## 性能要求

调试输出也应考虑大输入场景：

- 默认输出必须是局部和过滤后的
- 默认不要一次性打印整个 `10^5` 指令级 proto
- JSON 输出支持按范围截取
- 跨层查询依赖缓存映射，不临时全表扫描

## Consumes / Provides

- Consumes: 各层稳定 id、映射关系、分析事实
- Provides: 可读、可过滤、可对照的开发调试视图
