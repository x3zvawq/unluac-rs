# Test

## 职责

测试体系负责维护两类契约：

- `(case, dialect)` 的健康度
- 已修复问题的回归保护

## 正式入口

### 单元测试

- `cargo unit-test --jobs 8`
- 等价入口：`cargo lua test-unit`

### 校验基线

- `cargo clippy --all-targets --all-features --locked -- -D warnings`
- `cargo unit-test --jobs 8`

## 测试分层

### 单元测试

单元测试的最小执行单元是 `(case, dialect)`。

它回答：

- case 本身在该 dialect 是否健康
- 主反编译流水线在该 `(case, dialect)` 上是否健康

### 回归测试

回归测试负责锁定某个明确问题或决策，不承担全量健康扫描职责。

## 维护规范

### 1. 健康度测试看行为，不看文本相似

单元测试以：

- `exit status`
- `stdout`
- `stderr`

为正式健康标准，而不是源码逐字符一致。

### 2. 回归测试围绕具体问题组织

回归测试可以断言：

- 某层 dump 形状
- 某个恢复结果
- 某个错误边界

但每个回归测试都应指向明确问题。

### 3. pass 级测试放同目录 `tests.rs`

局部不变量测试与 pass 级测试应放在：

- 对应模块同目录 `tests.rs`

端到端行为继续放回统一测试体系，不把实现文件底部塞满测试。

### 4. 不做静默跳过与宽松兜底

如果某个 `(case, dialect)` 不在支持范围内，应从 manifest 层排除，而不是让运行时自己猜着跳过。

## 常用维护动作

### 改业务代码后

至少执行：

- `cargo clippy --all-targets --all-features --locked -- -D warnings`
- `cargo unit-test --jobs 8`

### 改单个问题时

优先做：

1. 跑对应 `(case, dialect)` 的单元测试
2. 如果问题有明确形状，再加回归测试

### 改 pass / helper 时

优先补：

- 同目录 `tests.rs` 的局部测试
- 必要时再补一个最小回归 case

## 维护检查清单

修改测试或新增 case 时，应至少检查：

1. 这是健康度测试，还是问题回归测试。
2. 失败信号是否能直接定位到 `(case, dialect)` 或明确问题。
3. 是否把本应进入 manifest 的支持范围问题错误地下沉到了运行时跳过。
4. pass 级测试是否仍位于同目录 `tests.rs`。
