# 测试体系

> 本文记录仓库的测试分层、运行方式、Lua case 规范与失败排查流程。

测试体系只保留两类公开测试：**单元测试** 与 **回归测试**。两类测试共用同一条端到端验证流水线，不再保留 smoke test、`src/` 内嵌测试、`tests/regression/` 快照测试，也不支持 bytecode-only case。

## 测试分层

| 层次 | 路径 / 入口 | 说明 |
| --- | --- | --- |
| 单元测试 | `tests/unit-case/` 下的 `.lua` | 覆盖结构清晰、语义明确的常规 Lua 源码样例。一个文件可以包含多个 case，每个 case 用函数包裹并在文件尾部统一调用输出。 |
| 回归测试 | `tests/regress-case/` 下的 `.lua` | 固定已经修复或正在定位的最小复现。每个文件应只承载同一类问题，避免把大块历史样例重新堆成不可维护集合。 |
| 测试驱动器 | `xtask/src/unit_test.rs` | `cargo unit-test` 的入口，负责任务筛选、并发调度、超时、失败汇总。 |
| 测试支持库 | `packages/unluac-test-support/` | 负责 Lua toolchain 调用、case manifest、反编译验证流水线和失败详情渲染。 |

`src/` 只保留实现代码，不放 `#[cfg(test)]`、`#[test]` 或模块同目录 `tests.rs`。如果某个内部不变量值得长期保护，应提炼成一个最小 Lua case，并让完整 pipeline 证明它不会退化。

## 验证流水线

每个 Lua case 都走同一条流程：

```text
源码执行
  -> 源码编译
  -> 编译产物执行并与源码输出对比
  -> 反编译编译产物
  -> 编译反编译结果
  -> 执行反编译结果并与源码输出对比
  -> 再次编译反编译结果
  -> 再次反编译
  -> 再次编译执行并验证输出
  -> 检查生成源码的文本收敛性与可读性断言
```

这条流水线的重点是验证真实用户路径：官方 toolchain 从源码产出字节码，`unluac` 反编译字节码，再证明生成源码在语义和稳定性上成立。

## 运行命令

```sh
# 全量测试，默认同时覆盖 unit 和 regression
cargo unit-test --jobs 8

# 只运行单元测试
cargo unit-test --suite unit --jobs 8

# 只运行回归测试
cargo unit-test --suite regression --jobs 8

# 只运行路径中包含某个片段的 case
cargo unit-test --case-filter common_05_boolean_expr

# 输出完整失败详情
cargo unit-test --output verbose --case-filter regress_01

# 指定再次编译/反编译轮数
cargo unit-test --recompile-rounds 1
```

`cargo test --test regression` 和 `tests/regression/` 不再作为测试入口存在。

## Lua Case 规范

每个 case 文件必须是源码文件，由仓库内 Lua toolchain 编译。测试体系不接收只提交 `.luac` / `.luau` 的 bytecode-only case。

推荐结构：

```lua
-- common_01_basics#1: 多赋值与局部变量初始化
local function test_assignments()
    local a, b, c = "alpha", 42, true
    local x, y, z = 1, 2, 3

    print("common_01_basics#1", a, b, c, x + y + z)
end

test_assignments()
```

约定：

- 每个子 case 前用 Lua 注释说明关注点，格式推荐为 `<file>#<number>: <说明>`。
- 每个子 case 的 `print` 首参使用同一个 `<file>#<number>` 标签，便于失败时定位到具体 proto / 子 case。
- 文件尾部统一调用所有子 case，避免 case 定义和执行顺序分散。
- 常规样例放入 `tests/unit-case/`，文件名使用 `common_` / `lua51_` / `luajit_` 等前缀。
- 回归样例放入 `tests/regress-case/`，文件名使用 `regress_` 前缀。
- 新增回归样例时优先最小化源码，避免复制几千行真实文件。

## 可读性断言

语义输出无法覆盖所有质量问题。某些问题只体现在反编译结果是否可读、是否生成了理想结构、是否避免了退化写法。此类检查通过 Lua 注释中的机器可读指令表达。

初始协议只保留少量稳定指令：

```lua
-- unluac: expect-contains [[local function test_assignments]]
-- unluac: expect-not-contains [[goto]]
-- unluac: expect-order [[if flag then]] [[else]]
```

含义：

| 指令 | 检查对象 | 说明 |
| --- | --- | --- |
| `expect-contains` | 生成源码 | 要求生成源码包含指定片段。 |
| `expect-not-contains` | 生成源码 | 要求生成源码不包含指定片段。 |
| `expect-order` | 生成源码 | 要求前一个片段出现在后一个片段之前。 |

这些注释只服务测试 runner，不能依赖 Lua 编译器保留注释。runner 必须在编译源码前解析它们，并在反编译生成源码后执行检查。

## 新增单元测试流程

1. 在 `tests/unit-case/` 新增或扩展 `.lua` 文件。
2. 为每个子 case 写清楚 Lua 注释说明与输出标签。
3. 如问题涉及可读性，在 case 前添加 `unluac:` 断言。
4. 运行 `cargo unit-test --suite unit --case-filter <file-stem>`。
5. 检查失败详情中的源码产物、chunk 产物与生成源码，确认测试表达的是稳定需求。

## 新增回归测试流程

1. 从问题样例中提取最小源码复现，放入 `tests/regress-case/`，文件名使用 `regress_XX_<topic>.lua`。
2. 保证 case 能通过仓库官方 toolchain 编译，不提交 bytecode-only 输入。
3. 用输出标签定位每个子 case；如果失败是可读性问题，必须补充 `unluac:` 断言。
4. 运行 `cargo unit-test --suite regression --case-filter <file-stem>`。
5. 修复根因后保留该 regression case，防止后续重构改回旧问题。

## 调试失败 Case 流程

1. 运行 `cargo unit-test --output verbose --case-filter <case>` 查看失败分类、产物路径和生成源码。
2. 先判断失败属于源码健康、反编译失败、生成源码编译失败、输出不一致、再次反编译不收敛，还是可读性断言失败。
3. 使用 `--stop-after`、`--dump-pass` 或相关 debug dump 定位最早丢失事实的层次，见 `10.debugging.md`。
4. 在根因层修复，不在后层用特判或兜底掩盖前层事实缺失。
5. 重跑对应 case，再跑 `cargo unit-test --jobs 8`。

## Lua 执行环境

`lua/` 目录提供完整 Lua 执行环境，用于编译和运行测试样例：

- `lua/build/lua5.1` 至 `lua5.5` / `luajit` / `luau`
- `lua/sources/` 对应源码
- 详见 `lua/README.md`
