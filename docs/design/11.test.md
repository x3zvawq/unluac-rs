# 测试体系

> 本文记录仓库的完整测试层次、运行方式及新增测试的规范。

## 测试层次

| 层次 | 路径 | 说明 |
|---|---|---|
| Unit tests（pass/模块级） | `src/*/tests.rs` | 各模块的局部不变量测试 |
| Regression（端到端） | `tests/regression/` | .luac → 反编译结果的端到端快照测试 |
| Regression runner | `tests/regression.rs` | 驱动器，枚举 `tests/regression/` 下所有 case |
| Lua test cases | `tests/lua_cases/` | Lua 源码 + 编译产物，供 regression 使用 |

## 运行命令

```sh
# 全量单元测试（并行 8 任务）
cargo unit-test --jobs 8

# 所有端到端测试
cargo test --test regression

# 仅运行匹配名字的 regression case
cargo test --test regression <substring>

# 更新 regression 快照
cargo test --test regression -- --update

# 运行 clippy 全量检查
cargo clippy --workspace --all-targets --all-features --locked -- -D warnings
```

## 不能用 `cargo test`？

`cargo unit-test` 是 `xtask/` 里的自定义命令，等价于：
```sh
cargo test --lib --workspace
```

## Regression 快照格式

每个 case 目录（`tests/regression/case-name/`）包含：
- `input.luac` 或 `input.lua`（被测文件）
- `expected.lua`（期望输出，提交进 git）
- `config.toml`（可选，override 默认解码参数）

运行 `--update` 会重新生成 `expected.lua`；正常运行时对比字符串 diff。

## 单元测试规范

- **不内嵌到实现文件底部**：测试放在同目录 `tests.rs` 文件中，实现文件只保留：
  ```rust
  #[cfg(test)]
  mod tests;
  ```
- 测试文件第一行应是模块级注释，说明测试的关注点。

## 新增 Regression Case 流程

1. 确认 Lua 源码在对应 Lua 版本下可编译：`lua5.4 -c xxx.lua`
2. 把 `.lua` 或 `.luac` 放到 `tests/lua_cases/` 或 `tests/regression/case-name/input.luac`
3. 运行 `cargo test --test regression -- --update` 生成期望快照
4. 检查 `expected.lua` 内容，确认语义正确
5. Commit 两个文件

## 调试失败 Case 流程

1. 运行 `cargo test --test regression <case-name>` 看 diff
2. 利用 `--target-stage` 或 `--dump-pass` 定位层次（见 `10.debugging.md`）
3. 在根因层修复；修复后重跑 `cargo test --test regression` 确认不引入新失败

## 当前状态说明

六个预存失败 case（`known_failures`）在 `tests/regression.rs` 中标注。  
每一轮修改后应确保不新增失败；如能顺手修复已知失败也应同步删除标注。

## Lua 执行环境

`lua/` 目录提供完整的 Lua 执行环境，用于编译测试样例：
- `lua/build/lua5.1` 至 `lua5.5` / `luajit` / `luau`
- `lua/sources/` 对应源码（用于本地重新编译）
- 详见 `lua/README.md`
