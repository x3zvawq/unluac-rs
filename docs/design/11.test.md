# Test

## 目标

测试体系负责回答两件事：

- 当前支持范围内的 `(case, dialect)` 是否真的健康
- 某个已经修好的问题，后续修改有没有把它重新搞坏

这里不把“测试数量更多”当成目标，而是把“职责边界清晰、失败信号可定位”当成目标。测试失败应该直接指向某个 case、dialect 或既有修复点，而不是依赖兜底逻辑把真实问题吞掉。

## 总体分类

项目测试体系固定分成两类：

1. 单元测试
2. 回归测试

其中单元测试的最小执行单元是 `(case, dialect)`。

## 单元测试

单元测试通过 `cargo unit-test` 执行，等价入口是 `cargo lua test-unit`。它负责对所有已纳入支持范围的 `(case, dialect)` 做统一健康检查。

当前实现里，单元测试包含两个 suite：

- `case-health`
- `decompile-pipeline-health`

它们不是顶层测试分类，而是同一套单元测试入口内部的两个健康检查 suite。

### case-health

`case-health` 负责证明“case 自己在对应 dialect 下是健康的”。

每一个 `(case, dialect)` 进入 `case-health` 后，至少必须通过下面这些检查：

1. 使用 `lua/build/<dialect>/lua` 直接解释原始源码，进程必须成功退出
2. 使用 `lua/build/<dialect>/luac` 编译原始源码，编译必须成功
3. 使用同一 dialect 的 `lua` 执行编译后的 chunk，进程必须成功退出
4. 原始源码直跑得到的 `(exit status, stdout, stderr)` 必须与原始源码编译后执行 chunk 的结果完全一致

### decompile-pipeline-health

`decompile-pipeline-health` 负责证明“主反编译流水线当前是健康的”。

每一个 `(case, dialect)` 进入 `decompile-pipeline-health` 后，至少必须通过下面这些检查：

1. 先通过同一 `(case, dialect)` 的 `case-health` 基线检查
2. 把编译后的 chunk 输入反编译主入口，并要求成功走到最终源码输出阶段
3. 把反编译得到的源码重新交给对应 dialect 的 `luac` 编译，确保输出是合法 Lua 源码
4. 用对应 dialect 的 `lua` 执行反编译后的源码，进程必须成功退出
5. 反编译源码执行得到的 `(exit status, stdout, stderr)` 必须与原始源码基线完全一致

### 入口参数

当前 `cargo unit-test` 支持下面这些参数：

- `--suite <all|case-health|decompile-pipeline-health>`
- `--dialect <all|lua5.1|lua5.2|lua5.3|lua5.4|lua5.5>`
- `--case-filter <substring>`
  可重复传入；多次传入时按“任一匹配”处理
- `--output <simple|verbose>`
- `--timeout-seconds <n>`
- `--progress <auto|on|off>`
- `--color <auto|always|never>`
- `--jobs <n>`

当前还支持下面这些环境变量：

- `UNLUAC_TEST_OUTPUT=simple|verbose`
- `UNLUAC_TEST_PROGRESS=auto|on|off`
- `UNLUAC_TEST_COLOR=auto|always|never`

当前默认值：

- `suite=all`
- `dialect=all`
- `output=simple`
- `timeout-seconds=10`
- `progress=auto`
- `color=auto`
- `jobs=1`

规则：

- 对输出使用精确比较，不做模糊归一化，不吞掉 stderr
- 单元测试的通过条件是“最终可执行且语义等价”，不是“中间 dump 看起来差不多”
- 不要求生成源码与原始源码逐字符一致；源码文本是实现细节，行为等价才是健康标准
- 失败必须直接归因到具体的 `(case, dialect)`，并给出可定位的信息

## 回归测试

回归测试只做一件事：把某个已经修好的 bug、优化结果或语义决策固定下来，防止后续修改回退。

回归测试的典型特点：

- 必须围绕一个明确问题组织，而不是笼统扫全量 case
- 允许断言中间层形状，例如某个阶段的 dump、结构恢复结果或特定错误语义
- 可以只覆盖一个 case，甚至只覆盖一个 case 的一个局部模式

回归测试不承担的职责：

- 不负责提供 case corpus 的全量健康度
- 不负责给所有 case 建立统一的执行输出基线
- 不应该把“当前错误但暂时能接受”的行为大量固化成长期契约

## case 与 dialect 矩阵

测试矩阵由 manifest 集中维护，最小执行单元始终是 `(case, dialect)`。

规则：

- 哪些 case 被纳入单元测试，必须显式维护
- 每个 case 支持哪些 dialect，必须显式维护
- `common` case 可以声明多个 dialect
- dialect-specific case 只声明对应支持范围内的 dialect
- 回归测试可以继续只引用某个具体 `(case, dialect)` 组合

## 准入与失败策略

任何 case 进入测试体系，都应遵守显式准入，而不是隐式“顺手跑一下”。

失败策略也必须直接：

- 单元测试失败，说明 case 本身、vendored toolchain 或主反编译流水线存在真实问题
- 回归测试失败，说明某个已经固定下来的修复点被回退

不要在测试里做以下事情：

- 自动吞掉 stderr
- 自动重试后把第一次失败覆盖掉
- 因为某个组合暂时不支持就静默忽略

如果某个组合当前不该纳入支持范围，正确做法是不要把它写进 manifest，而不是让测试在运行时自己猜着跳过。
