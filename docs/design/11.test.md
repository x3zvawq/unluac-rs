# Test

## 目标

测试体系负责回答三件事：

- case 自己是不是健康的
- 当前反编译主流程是不是还能把已纳入支持范围的 case 正确跑通
- 某个已经修好的问题，后续修改有没有把它重新搞坏

这里不把“测试数量更多”当成目标，而是把“职责边界清晰、失败信号可定位”当成目标。测试不应该用大量兜底逻辑把真实问题吞掉；相反，应该让失败直接指向 case、dialect、stage 或某个既有修复点。

## 总体分类

从职责上，项目测试体系固定分成两大类：

1. 单元测试
2. 回归测试

其中“单元测试”在这里不是狭义的“只测一个函数”，而是“每一个被纳入支持范围的最小测试单元都必须有稳定健康信号”。对本项目来说，这个最小单元通常是 `(case, dialect)`。

## 单元测试

单元测试分成两部分：

- `case-health`：固定 case 本身的健康度
- `decompile-pipeline-health`：固定主反编译流水线的端到端健康度

### case-health

`case-health` 负责证明“case 自己没有问题”，它只回答 case 和宿主 dialect 的关系，不回答反编译器内部某一层怎么实现。

每一个 `(case, dialect)` 进入 `case-health` 后，至少必须通过下面四步：

1. 使用 `lua/build/<dialect>/lua` 直接解释原始源码，进程必须成功退出
2. 使用 `lua/build/<dialect>/luac` 编译原始源码，编译必须成功
3. 使用同一 dialect 的 `lua` 执行编译后的 chunk，进程必须成功退出
4. “源码直跑”的 `(exit status, stdout, stderr)` 必须和“源码先编译再执行 chunk”的结果完全一致

规则：

- 对输出使用精确比较，不做模糊归一化，不吞掉 stderr
- case 一旦进入 `case-health`，失败就是 case 或 toolchain 的真实问题，不能通过“先跳过再说”来兜底
- 如果某个 case 需要额外前置条件，必须在 manifest 里显式表达，而不是在测试里临时猜

`case-health` 的产物是每个 `(case, dialect)` 的可信基线。后续 `decompile-pipeline-health` 必须直接复用这条基线，而不是再各自发明“差不多就算通过”的比较规则。

### decompile-pipeline-health

`decompile-pipeline-health` 负责证明“主反编译流水线当前是健康的”。

每一个 `(case, dialect)` 进入 `decompile-pipeline-health` 后，至少必须通过下面五步：

1. 用对应 dialect 的 `luac` 把原始源码编译成 chunk
2. 把 chunk 输入反编译主入口，并要求成功走到最终源码输出阶段
3. 把反编译得到的源码重新交给对应 dialect 的 `luac` 编译，确保输出是合法 Lua 源码
4. 用对应 dialect 的 `lua` 执行反编译后的源码，进程必须成功退出
5. 反编译源码执行得到的 `(exit status, stdout, stderr)` 必须与 `case-health` 基线完全一致

规则：

- `decompile-pipeline-health` 的通过条件是“最终可执行且语义等价”，不是“中间 dump 看起来差不多”
- 不要求生成源码与原始源码逐字符一致；源码文本是实现细节，行为等价才是健康标准
- 如果某个 case 还不能稳定走到最终输出阶段，就不要把它纳入这个 suite；准入由 manifest 显式控制

## 回归测试

回归测试只做一件事：把某个已经修好的 bug、优化结果或语义决策固定下来，防止后续修改回退。

回归测试的典型特点：

- 必须围绕一个明确问题组织，而不是笼统扫全量 case
- 允许断言中间层形状，例如 HIR、AST、Readability dump 或特定错误语义
- 允许只覆盖一个 case，甚至只覆盖一个 case 的一个局部模式

回归测试不应该承担的职责：

- 不负责提供 case corpus 的全量健康度
- 不负责给所有 case 建立统一的执行输出基线
- 不应该把“当前错误但暂时能接受”的行为大量固化成长期契约

一条修复进入回归测试后，应该同时满足两点：

- 有一个最小但稳定的 regression fixture，精确描述这次修复保护什么
- 如果这个修复已经把某个 case 提升到“可稳定支持”，该 case 还应该被纳入相应的 unit health suite，而不是只留在 regression

## case 与 dialect 矩阵

后续 manifest 不应继续使用“一个 case 只绑一个 dialect”的模型，而应升级成“一个 case 可以声明多个目标 dialect”的矩阵模型。

建议把 case 分成两类：

- `common` case：语义上设计为多 dialect 共用，但实际跑哪些 dialect 仍然必须显式列出
- dialect-specific case：只跑所属 dialect

规则：

- 不因为文件放在 `common/` 下，就自动默认跑所有 dialect
- 哪些 dialect 已经被该 case 正式支持，必须在 manifest 里写明
- `case-health` 和 `decompile-pipeline-health` 都以 `(case, dialect)` 为最小执行单元
- regression 测试可以继续只引用某个具体 `(case, dialect)` 组合

这样可以避免两种极端：

- 一边是每个 common case 只跑一个 dialect，覆盖率明显不足
- 另一边是默认对所有 dialect 做笛卡尔积，导致不成熟组合混进正式测试

## 准入与失败策略

任何 case 进入测试体系，都应遵守显式准入，而不是隐式“顺手跑一下”：

- 是否进入 `case-health`
- 是否进入 `decompile-pipeline-health`
- 需要跑哪些 dialect

这些信息都应该由 manifest 集中维护。

失败策略也必须直接：

- `case-health` 失败，说明 case 自身或 vendored toolchain 有问题
- `decompile-pipeline-health` 失败，说明主反编译流水线最终语义不成立
- regression 失败，说明某个已经固定下来的修复点被回退

不要在测试里做以下事情：

- 自动吞掉 stderr
- 自动重试后把第一次失败覆盖掉
- 因为某个 dialect 暂时不支持就静默忽略

如果某个组合当前不该纳入支持范围，正确做法是不要让它进入 manifest 对应 suite，而不是让测试运行时自己猜着跳过。
