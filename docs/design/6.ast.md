# AST

## 目标

AST 是 target-dialect-aware 的语法树。

这层不再负责恢复决策，而是负责把 HIR 中已经做出的结构与表达式决策，落到某个目标 Lua dialect 真正支持的语法节点上。

## 当前规则

- AST 构建必须带明确的 target dialect 上下文
- AST 可以比 HIR 更保守，但不能比 HIR 更激进
- AST 不直接持有最终字符串命名
- AST 不重新做 CFG、GraphFacts、Dataflow 或 Structure 恢复
- AST 允许使用 Lua 语法糖节点
- 最终 AST 不保留 `Unresolved / Unstructured` 节点
- AST lowering 可以先保守地产出合法语法
- AST 逻辑上按函数嵌套形成树
- AST lowering 不负责消费 `Decision`；这类共享决策 DAG 必须在 HIR 末端已经被消掉
- AST lowering 不能把 HIR 的 `TableSetList` 原样下传；必须在 AST 阶段之前或 lowering 过程中把它折回构造器，或者改写成合法的逐项赋值
- AST 的方言约束以 “target dialect + capability set” 表达，而不是给每个 AST 节点重复挂 version tag
- HIR 中的 `ToBeClosed / Close` 只是 lowering 证据，不是最终 AST 的保留节点
- Lua 5.5 `global` 这类方言语法也属于 AST 的职责边界，而不是 HIR 的职责边界

## 目标方言约束

AST lowering 需要根据 target dialect 处理：

- 是否支持 `goto`
- 是否支持 label
- 是否支持 `continue`
- 是否支持 `local <const>`
- 是否支持 `local <close>`
- 是否支持 `global`
- 是否支持 `global<const>`
- 某些字符串、数字、作用域和局部声明形式是否允许
- 某些语法糖是否可以直接使用

HIR 到 AST 的 lowering 必须显式处理这些约束，而不是把能力不兼容的问题留到代码生成阶段。

## 方言能力模型

AST 不建议给每个节点都挂一个 dialect tag。

更合适的做法是：

- lowering 输入携带 `AstTargetDialect`
- `AstTargetDialect` 提供当前目标方言的 capability 集
- AST 节点本身只表达“已经被允许使用的语法”

这样做的原因是：

- 节点 payload 更小，不重复记录版本信息
- 语法合法性检查集中在 lowering / validation 阶段
- 同一套 AST 节点可以服务多个相邻 Lua dialect
- 后续新增 dialect 时，只需要扩 capability 映射，而不是把 version 判断散落到所有节点

可以先按下面的概念建模：

```text
AstTargetDialect
  version
  caps: AstDialectCaps

AstDialectCaps
  goto_label
  continue_stmt
  local_const
  local_close
  global_decl
  global_const

AstDialectFeature
  GotoLabel
  ContinueStmt
  LocalConst
  LocalClose
  GlobalDecl
  GlobalConst
```

这里的 `AstDialectFeature` 更适合作为 lowering / validation 时的规则枚举；
`AstDialectCaps` 是某个具体 target dialect 对这些 feature 的实例化结果。

## 当前边界

- HIR 决定结构
- AST 决定语法
- Readability 决定折叠与语法糖收敛
- Naming 决定最终字符串命名
- Generate 决定文本输出

这三层边界固定后：

- AST 不回头修改 HIR 的恢复决策
- Generate 不回头修改 AST 的语法决策

## 标识符边界

AST 中的变量、参数、upvalue 和全局引用在这一层仍然保持为符号引用或绑定引用，不直接绑定为最终字符串。

这样做的原因是：

- AST 阶段之后仍会继续做可读性折叠
- 某些临时绑定会在折叠后消失
- 如果过早分配字符串名称，会造成跳号、残留命名和低质量输出

最终字符串命名统一放在 Naming 阶段完成。

## 结构边界

- HIR 继续按平行 proto 保存，便于分析
- AST 在逻辑上转成嵌套函数树，便于语法化、命名和代码生成
- 闭包在 AST 中表现为嵌套函数表达式或函数声明，而不再强调 proto 平行存储模型

## 函数语法糖

AST 允许显式建模 Lua 的函数声明语法糖：

- `FunctionDecl`
- `LocalFunctionDecl`

并区分：

- `function a.b.c()`
- `function a.b:c()`

也就是说，AST 中的方法声明语法不是通过普通赋值和函数表达式再现，而是作为真实语法节点保留。

## 局部声明属性

Lua 的局部声明语法并不完全等价，AST 需要显式区分绑定属性，而不是把所有局部变量都压成统一的 `local`。

当前至少需要覆盖：

- 普通局部：`local x = ...`
- 常量局部：`local x <const> = ...`
- to-be-closed 局部：`local x <close> = ...`

这意味着 AST 里的局部声明需要拆成：

- 局部声明语句本身
- 每个 binding 的属性

对应契约是：

- `AstLocalAttr::None` 表示普通局部
- `AstLocalAttr::Const` 只允许在 target dialect 支持 `local <const>` 时出现
- `AstLocalAttr::Close` 只允许在 target dialect 支持 `local <close>` 时出现
- 如果上游证据不足以证明某个 binding 语义上就是 `<const>`，AST 必须保守回退成普通 `local`
- `<close>` 不是“可选语法糖”；如果语义需要 to-be-closed 行为，AST 必须保留这层语义，或者在目标方言不支持时明确失败，不能静默降级

另外，HIR 中仍可能残留 `TempId`。为了保证 AST 第一版在语法上仍然合法：

- AST 允许显式声明 temp binding
- 这类 temp declaration 属于“为保证合法语法而引入的保守声明”，不是源码级变量识别结论
- 后续 Readability / Naming 可以继续把它们合并、消除或改名

也就是说，AST 里的局部声明 binding 不应只允许 `LocalId`，还要允许 `TempId`

## goto + close 契约

`goto + close` 这类 case 的难点不在于 `goto` 本身，而在于跳转跨越了活跃的 to-be-closed 作用域。

这里需要把几个层次分开：

- HIR 里的 `ToBeClosed` 表示某个局部 binding 从这一点开始进入 active close scope
- HIR 里的 `Close` 表示存在一次“离开 active close scope”的语义事件
- 最终 AST 不直接保留 `Close` 语句节点

AST lowering 需要满足：

- 如果 `goto / break / return` 离开了仍然活跃的 close scope，最终 AST 必须用合法的词法结构表达这件事
- 这通常意味着 AST 需要恢复出合适的 `DoBlock`、loop body 或函数 body 边界，并在对应 block 中保留 `local <close>`
- `Close` 在 AST lowering 中更像“必须恢复词法退出边界”的证据，而不是最终语法节点
- 不能把这类语义静默擦掉，也不能生成一个“裸 close 语句”去假装 Lua 源码里存在这类语法

因此，`goto + close` 的 lowering 契约可以概括成：

- HIR 可以显式携带 `ToBeClosed / Close`
- AST 必须把它们消费成合法的词法块结构
- Generate 阶段不再看到 `Close`

当前更细的 lowering 前置条件可以写成：

- 如果 `ToBeClosed` 直接对应某个刚建立的局部 binding，AST 可以把该 binding 标成 `AstLocalAttr::Close`
- 如果 `ToBeClosed` 对应的是一个尚未语法化的 temp binding，AST 可以显式把这个 temp 提升成保守的局部声明，再附上 `Close` 属性
- 如果 HIR 里仍残留显式 `Close`，AST 要么恢复出能够吸收它的词法块边界，要么明确报 lowering 失败；不能把 `Close` 原样泄漏给 Generate

## global 契约

Lua 5.5 的 `global` 语法不只是变量访问方式不同，还携带声明与初始化约束。

AST 需要区分：

- 普通全局初始化：`global x = ...`
- 常量全局初始化：`global<const> x = ...`
- global 列表声明：`global x, y = ...`

这里的约束是：

- `global` 只允许在 `global_decl = true` 的 target dialect 下出现
- `global<const>` 只允许在 `global_const = true` 的 target dialect 下出现
- 当前 HIR 里最稳定的 global-decl 证据是 `ErrNil + global assign` 组合；AST 至少要能消费这类形状
- 如果上游没有提供足够证据证明某次全局写入语义上就是 `global` 声明，AST 不能为了“更像源码”而擅自把普通全局赋值改写成 `global`
- 某些零运行时痕迹的 `global` 声明如果在前层没有留下证据，AST 也不能凭空发明；这属于“信息已在前层丢失”的问题，而不是 AST 自行补猜的空间

## Fallback 边界

- HIR 可以保留少量 `Unresolved / Unstructured` fallback
- AST lowering 过程中允许短暂存在保守中间态
- 最终 AST 必须收敛成合法的 target-dialect-aware 语法树
- 如果 AST lowering 结束仍残留 unresolved 节点，应视为 lowering 失败或需要回退到更保守的合法 AST

## AST 之后的优化框架

AST 之后仍然存在一个独立的 `Readability` 阶段。

该阶段采用分阶段 fixed-point pass，而不是单个无限制的大循环。

这样可以保证：

- HIR simplify 只负责语义整理
- AST lowering 先产出合法 AST
- 语法糖和构造器折叠在 AST 上进行
- 命名始终基于最终保留下来的绑定集合

### 节点规则

- `AstExpr::Var` 使用 `AstNameRef`，不直接持有最终字符串
- `AstNameRef` 允许 `Temp(TempId)`，因为 HIR simplify 和 Readability 之后仍可能保留少量 temp 绑定
- `AstLocalDecl` 的 binding 允许 `Temp(TempId)`，用于把 HIR 残留 temp 语法化成合法局部变量
- `FieldAccess` 和 `IndexAccess` 在 AST 中显式区分
- `Call` 和 `MethodCall` 在 AST 中显式区分
- `FunctionDecl`、`LocalFunctionDecl` 和 `FunctionExpr` 显式区分
- `LocalDecl` 需要把 binding 和 binding attribute 显式分开，不能只保留一组局部名
- `GlobalDecl` 需要把 global binding 和 global attribute 显式分开，不能把 `ErrNil` 继续当成 AST 语义节点
- `AstBlock` 只保存语句列表，不再承载恢复 metadata
- `AstReturn.values` 直接是表达式列表，Lua 的多值语义由表达式位置承担
- AST 不应该再看到 `HirExpr::Decision`；如果进入 AST lowering 时仍残留这类节点，应视为 HIR 契约违背
- HIR 中的 `TableSetList` 也不是 AST 的最终节点；AST 需要把这类“数组段批量写入”恢复成目标方言可打印的合法结构
- HIR 的 `NumericFor.binding/start/limit/step` 需要直接映射到 AST 的 numeric for 语法，
  不应该在 AST 层重新回头推导 VM 级控制寄存器
- `AstTargetDialect / AstDialectCaps` 是 lowering 和 validation 的上下文，不需要复制到每个 AST 节点
- `AstStmt::Goto / AstStmt::Label` 只能在 `goto_label = true` 的 target dialect 下出现
- HIR 中的 `Continue` 到 AST 时必须显式看 target dialect：
  支持 `continue` 的 dialect 可以直接保留成 `AstStmt::Continue`；
  不支持的 dialect 需要在 AST lowering 阶段改写成合法的 `goto + label` 或等价结构
- `AstLocalAttr::Const` 只能在 `local_const = true` 的 target dialect 下出现
- `AstLocalAttr::Close` 只能在 `local_close = true` 的 target dialect 下出现
- `AstStmt::GlobalDecl` 只能在 `global_decl = true` 的 target dialect 下出现
- `AstGlobalAttr::Const` 只能在 `global_const = true` 的 target dialect 下出现
- HIR 中的 `ToBeClosed / Close` 不是 AST 最终节点；AST 需要把它们吸收到合法的块作用域和 `local <close>` 语法中

## Consumes / Provides

- Consumes: `HirModule`, target dialect context
- Provides: `AstModule`
