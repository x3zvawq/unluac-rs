# AST

## 职责

AST 负责把 HIR 落成合法 Lua 语法树，并显式保存目标方言允许的语法形状。

这一层回答的问题是：

- 这段 HIR 应该落成哪种合法 Lua 语法节点
- 目标 dialect 是否允许这种语法
- 哪些语义需要保守表达而不能在这里继续糖化

这一层不负责：

- 追求“更像人写源码”的最终形状
- 最终命名
- 文本布局

## 应消费的前层事实

AST 应消费：

- `HirModule`
- `AstTargetDialect`
- HIR 已经给定的结构与 binding 身份

AST 不应回头做：

- 新的结构恢复
- 缺失 global / goto / close 语义的补猜
- 可读性优先的 sugar 选择

## 应优先复用的共享设施

### build 分析与 pattern

- `analysis.rs` 负责 HIR -> AST lowering 需要的分析
- `patterns/syntax.rs` 负责合法语法化
- `patterns/shape.rs` 只保留语义保真需要的形状规则

纯可读性 sugar 不应继续放在 `patterns/shape.rs`。

### AST 共享节点

`src/ast/common.rs` 里的节点与 capability 是正式契约，例如：

- `AstTargetDialect`
- `AstDialectCaps`
- `AstExpr::SingleValue`
- `AstLocalAttr`
- `AstGlobalAttr`
- `AstFunctionDecl`
- `AstLocalFunctionDecl`

如果某个语义需要跨 Readability / Generate 稳定存在，应优先表达成 AST 节点或属性，而不是靠临时机械壳。

## 维护规范

### 1. AST build 只负责合法语法化与语义保真

AST build 适合放：

- `goto + close` 的合法词法块表达
- `SingleValue` 这类语义保真节点
- target dialect capability 校验
- 必要的保守 local/global 语法节点
- 为了保守表达 residual temp 而临时 hoist 的空 local 声明
  - 但这些 hoisted temp 应按 block 里的首次出现顺序排布，避免后续 Readability 的
    `statement_merge` 因声明顺序和首次赋值顺序错位而无法把它们下沉回源码附近

AST build 不适合放：

- method / function sugar
- installer IIFE 美化
- field access sugar
- local merge / stmt merge

这些属于 Readability。

### 2. target dialect 能力通过 capability 表达

AST 节点本身不需要给每个节点挂版本 tag。

语法合法性应通过：

- `AstTargetDialect`
- `AstDialectCaps`

集中控制，而不是把版本判断散落到各节点处理代码。

### 3. binding 身份保持稳定

AST 仍应保留：

- `ParamId`
- `LocalId`
- `TempId`
- `UpvalueId`
- `AstSyntheticLocalId`

不要在 AST 阶段把它们提前改成最终字符串。

### 4. Generate 不替 AST 兜底

如果某个语法节点对 Generate 来说是必须显式存在的，例如：

- `global *`
- `global<const>`
- `local <close>`
- `SingleValue`

AST 就应明确建模，而不是让 Generate 再猜。

## 向后提供的事实

AST 向 Readability / Naming / Generate 正式提供：

- 合法 Lua 语法树
- 目标 dialect capability 下允许出现的节点
- 稳定 binding 身份
- 语义保真需要的保守节点与属性

## 维护检查清单

修改 AST 时，应至少检查：

1. 这是合法语法化问题，还是源码美化问题。
2. 该信息是不是应该先在 HIR 或更前层表达。
3. 这个语义是不是应该在 AST 成为显式节点或属性，而不是机械 temp/local 壳。
4. 这个改动会不会把 Readability 或 Generate 的职责拉回 AST。
